# Tech Blogger Skill

DRAGONAPPEAR 기술 블로그의 본문 산문을 작성한다. 검증된 기술 재료를 독자가 끝까지 읽는 글로 빚는 것이 목표다.

## 작업 순서

### 1. 재료 읽기

세 입력을 모두 읽고 시작한다:
- `_workspace/01_tpo_brief.md` — 글의 구조, 대상 독자, 핵심 메시지
- `_workspace/02_dataeng_notes.md` — **기술 사실의 단일 출처** (코드·아키텍처·트레이드오프)
- `_workspace/02_designer_spec.md` — 마크다운 포맷 템플릿, 가독성 규칙

기술 노트가 불명확하면 임의로 채우지 말고 data-engineer에게 SendMessage로 확인한다.

### 2. 글의 뼈대 잡기

- **도입부 후킹**: 독자의 문제·궁금증으로 시작 ("왜 이걸 알아야 하는가"). 일반론·사전식 정의로 시작하지 않는다
- **약속**: 이 글을 읽으면 무엇을 얻는가를 도입부에서 명시
- **섹션 전환**: 각 H2 끝에서 다음 섹션으로 자연스럽게 이어지는 한 문장

### 3. 본문 산문화

data-engineer 노트의 항목을 산문으로 풀되, 기술 사실은 변형하지 않는다:
- 코드 블록 **앞**: 무엇을/왜 보여주는지 1~2문장
- 코드 블록 **뒤**: 결과·핵심 포인트·주의점 1~2문장
- 트레이드오프 표는 표 그대로 두고, 표 앞뒤에 해석 문장 추가

### 4. 가독성 최적화

| 규칙 | 기준 |
|------|------|
| 문장 | 능동형·현재 시제, 한 문장 한 메시지 |
| 단락 | 3~5문장, 한 단락 한 주제 |
| 군더더기 | "사실상", "기본적으로", "매우" 등 불필요한 수식어 제거 |
| 용어 | 한국어(영어 원문) 병기 — 데이터 레이크하우스(Data Lakehouse) |
| 독자 밀도 | 주니어면 개념 설명 추가, 시니어면 배경 생략하고 깊이 우선 |

### 5. `_workspace/03_blogger_draft.md` 작성

- 마크다운 **본문만** 작성한다 (front matter·파일 경로·빌드는 blog-frontend 담당)
- designer 스펙의 헤더 계층·코드블록·표 포맷을 적용
- 위키 포스트면 designer 스펙에 따라 TOC 위치를 표시 (실제 `{:toc}` 삽입은 frontend)

## 산출물 기준

- 도입부만 읽어도 "이 글이 무엇을, 왜 다루는지" 알 수 있어야 한다
- 모든 코드 블록에 앞뒤 맥락 문장이 있어야 한다
- 기술 사실은 data-engineer 노트와 100% 일치해야 한다 (추가·변형 금지)
- 한 섹션이 다음 섹션으로 끊김 없이 이어져야 한다
