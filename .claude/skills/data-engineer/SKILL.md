# Data Engineer Skill

DRAGONAPPEAR 기술 블로그의 데이터 엔지니어링·클라우드 콘텐츠에 실무 깊이와 검증된 코드를 제공한다.

## 작업 순서

### 1. 입력 파악

- `_workspace/01_tpo_brief.md` — 아웃라인, 대상 독자, "기술 검증 포인트"
- 기존 관련 포스트가 있으면 `data/total-document-url-list.json`으로 확인해 중복·연결점 파악

### 2. 섹션별 기술 보강

아웃라인의 각 H2/H3에 대해 다음을 채운다:
- **핵심 메커니즘**: 어떻게 동작하는가 (내부 동작·데이터 흐름)
- **실무 관점**: 운영 시 주의점, 흔한 실패, 비용/성능 함정
- **트레이드오프**: 대안 대비 장단점 (비용·지연·처리량·운영부담)

### 3. 코드 예제 작성

- 실제 동작 수준으로 작성하고 버전·전제 조건을 명시한다
- 언어 식별자를 정확히 표기: ` ```sql `, ` ```python `, ` ```yaml `, ` ```bash `
- 예제마다 "무엇을 보여주는가" 한 줄을 앞에 남긴다 (tech-blogger가 맥락 문장으로 확장)

### 4. `_workspace/02_dataeng_notes.md` 작성

아래 구조를 따른다:

```markdown
# Data Engineer Notes: {포스트 제목}

## 정정 사항 (TPO 아웃라인 대비)
- {부정확했던 항목}: {올바른 내용 + 근거}

## 섹션별 기술 노트
### {H2 섹션}
- 핵심 메커니즘:
- 실무 관점:
- 트레이드오프:
- 코드 예제: (아래 코드 블록)

## 검증된 코드 예제
` ```python ... ``` `

## 트레이드오프 비교표
| 항목 | 옵션 A | 옵션 B | 권장 상황 |

## 검증 상태
- {항목}: 검증 완료 / 공식 문서 확인 필요({링크})
```

## 스택별 작성 가이드

도메인별 상세 패턴·코드 스니펫이 필요하면 해당 파일을 읽는다:

- GCP 주제 (BigQuery/Dataflow/Pub/Sub/Composer) → `references/gcp.md`
- 오픈소스 데이터 스택 (Spark/Airflow/Kafka/Flink/dbt/Iceberg) → `references/oss-data-stack.md`

## 산출물 기준

- 모든 기술 주장은 검증 가능해야 한다 — 불확실하면 "검증 필요"로 명시 (추측 금지)
- 코드는 복사해 실행 가능한 수준 (의사코드 지양, 불가피하면 명시)
- 아키텍처 설명은 "왜 이 선택인가"를 반드시 포함한다
- 기술 용어는 한국어(영어) 병기
