# Technical Writer Skill

재료를 독자가 끝까지 읽는 글로 바꾼다. 출력은 `_workspace/03_draft.md` — **마크다운 본문만**(front matter·빌드는 editor 담당).

## 작업 순서

### 1. 재료 읽기
`00_brief.md`(핵심 질문·아웃라인), `02_sme_notes.md`(기술), `01_research.md`(있으면, 출처·UNVERIFIED)를 읽는다. 핵심 질문에서 벗어난 재료는 쓰지 않는다.

### 2. 뼈대 잡기
brief 아웃라인을 따르되 **논리 흐름**으로 재배열한다: 구체적 문제/상황 → 왜 어려운가 → 어떻게 풀리는가 → 한계. 소제목은 내용을 말하게 한다("개요" ✗ → "메달리언이 멈추는 곳" ✓).

### 3. 본문 산문화 — DRAGONAPPEAR 보이스
- **문제로 연다.** 정의 나열이 아니라 구체적 문제·상황. 단 허구적 인물·시간순 스토리는 금지, 사실 진술로. (예: "silver.orders 한 행으로 답 못 하는 질문 셋".)
- **실제 데이터로 붙잡는다.** "고객 엔티티" ✗ → `canonical_customer_id = "CUST-029182"` ✓.
- **직설적으로 설명한다.** 극적 장치 없이 사실과 근거로. 단정을 피해야 할 곳은 헤지하되, 서스펜스로 끌지 않는다.
- **정직하게.** 트레이드오프·미해결을 숨기지 않는다. 과장은 신뢰를 깎는다.
- **SME 코드·수치는 보존.** 문장으로 감싸되 사실은 그대로. 의심되면 고치지 말고 메모.
- **UNVERIFIED는 헤지.** "정확한 수치는 버전에 따라 다르다" 식으로 단정 회피.
- 한국어 산문 + 영어 기술 용어.

#### 금지 — 소설체
기술 블로그는 정보 전달이 목적이다. 아래는 쓰지 않는다:
- **허구적 장면·인물 vignette** ("한참을 헤매다 깨닫는다", "팀원이 물었다") → 사실 직설로.
- **클리프행어·서스펜스** ("그런데 여기서 문제가 생긴다", "과연…") → 문제 바로 진술.
- **의인화·비유 동사** (무너지다/좌초하다/침묵하다/살아있다/올라타다/뛰다) → 직설 동사(탐지되지 않다/실패하다/위치하다/증가하다).
- **시리즈 반결산 콜백** ("N편을 지나왔다", 도입부 극적 재호출) → 담백한 요약.
- **감정적·수사적 마무리** (대구·선언조) → 기술적 결론만.

### 4. 가독성 최적화
- 스캔 가능한 구조. 긴 글이면 본문 맨 위 `<!-- toc -->` 마커.
- 코드블록 언어 식별자 유지. SME의 표·다이어그램을 다듬어 적재적소 배치.
- 한 문단 한 생각. 모바일에서 읽힌다고 가정(짧은 문단).

### 5. 위키링크
본문에서 다른 노드를 언급하면 `[[/{category}/{slug}]]`로 표기한다. editor가 brief의 intended relations와 대조한다.

### 6. `_workspace/03_draft.md` 작성
본문만. 재료가 비면 지어내지 말고 `[[SME 보강 필요: ...]]` 플레이스홀더를 남기고 SME에게 요청한다.

## 산출물 기준
- 구체적 문제 정의로 시작함(허구적 장면 아님)
- 핵심 질문 하나에 답함(곁가지 없음)
- 실제 예제 데이터로 추상을 붙잡음
- 한계·트레이드오프가 정직하게 드러남
- 소제목이 내용을 말함
- 소설체 없음(허구 장면·의인화 동사·클리프행어·"N편을 지나왔다" 콜백·수사적 마무리)
- front matter 없음(본문만)

## 후속/부분 재실행
`03_draft.md`가 있고 톤·구조 수정이면 다시 쓰되 사용자가 좋다고 한 부분은 보존한다.
