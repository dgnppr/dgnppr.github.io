# Blog Designer Skill

DRAGONAPPEAR 기술 블로그의 포스트 시각 구조와 사용성을 설계한다.

## 작업 순서

### 1. 현재 스타일 파악

다음을 읽어 기존 디자인 시스템을 파악한다:
- `_sass/_site-menu-bar.scss` — 네비게이션 스타일
- `_sass/_layout.scss` — 레이아웃 구조
- `_sass/_base.scss` — 베이스 스타일 (존재하는 경우)
- `_sass/_theme.scss` — 색상/테마 변수

### 2. TPO 브리프 분석

`_workspace/01_tpo_brief.md`를 읽어 포스트 유형과 구조를 파악한다:
- 코드 예제가 많은가? → 코드블록 포맷 강화
- 비교 분석인가? → 표(table) 활용 가이드
- 다이어그램이 필요한가? → 이미지 삽입 가이드라인

### 3. `_workspace/02_designer_spec.md` 작성

아래 구조를 따른다:

```markdown
# Designer Spec: {포스트 제목}

## 마크다운 포맷 템플릿

\`\`\`markdown
---
layout  : wiki
title   : {제목}
summary : {한 줄 요약}
date    : {YYYY-MM-DD} 00:00:00 +0900
updated : {YYYY-MM-DD} 00:00:00 +0900
tag     : {태그1} {태그2}
public  : true
parent  : {부모 카테고리 slug}
latex   : false
---

* TOC
{:toc}

## 도입부
...
\`\`\`

## 가독성 가이드라인

- 단락 길이: 최대 4~5문장
- H2 간격: 각 섹션은 충분한 내용을 포함 (최소 200자)
- 코드블록: 언어 식별자 필수 (```java, ```kotlin 등)
- 강조: **굵게**는 핵심 개념에만, 남용 금지

## 시각 요소 사용 기준

- 표: 3가지 이상 비교 시 사용
- 코드: 실행 가능한 예제 우선
- 이미지: alt 텍스트 필수

## UI 변경 요청 (해당 시)

| 항목 | 현재 상태 | 변경 요청 | 이유 |
|------|----------|----------|------|
| ... | ... | ... | ... |
```

### 4. 모바일/다크모드 체크

- `@media (max-width: 800px)` 기준 레이아웃 확인
- `_sass/_theme.scss`의 다크모드 변수와 호환되는지 확인
- 신규 CSS 추가 시 다크/라이트 모드 양쪽 대응을 명세에 포함

## 산출물 기준

- 마크다운 템플릿은 Frontend가 바로 사용할 수 있는 수준이어야 한다
- UI 변경 요청은 SCSS 선택자와 속성명을 구체적으로 명시한다
- 기존 클래스 재활용을 우선하고, 신규 클래스는 최소화한다
