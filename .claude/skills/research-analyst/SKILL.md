# Research Analyst Skill

기술 글의 신뢰도는 검증된 최신 사실에서 나온다. brief가 요청한 사실에만 집중해, 출처를 붙이고, 확인 못 한 건 솔직히 표시한다. 출력은 `_workspace/01_research.md`.

## 작업 순서

### 1. 검증 대상 파악
`_workspace/00_brief.md`의 "research-analyst에게" 절을 읽고 검증할 사실 목록을 뽑는다. 범위 밖은 건드리지 않는다.

### 2. 1차 출처 조사
출처 우선순위를 지킨다:
1. **공식 문서·릴리스 노트·가격표** — 벤더/프로젝트 공식 채널.
2. **독립 벤치마크·논문** — 인용 가능, 조건 명시된 것.
3. **신뢰할 블로그·컨퍼런스 발표** — 보조, 1차로 교차 확인.

`WebSearch`로 후보를 찾고 `WebFetch`로 1차 출처 본문을 확인한다. 큰 출력은 context-mode(`ctx_fetch_and_index`)로 인덱싱해 컨텍스트를 아낀다.

### 3. 주장-출처 매핑
각 사실을 표로 만든다: 주장 / 값 / 출처 URL / 출처 날짜 / 확신도(high·medium·low). **수치·버전을 절대 지어내지 않는다** — 이게 1원칙. 그럴듯한 거짓이 가장 해롭다.

### 4. 현행성·조건 확인
- 버전·기능·한도는 **오늘 기준** GA/최신을 확인한다. "예전엔 맞았던" 사실을 거르낸다.
- 벤치마크는 숫자만 옮기지 말고 **실행 환경·데이터 규모·날짜·독립여부**를 함께 적는다. 조건 없는 벤치마크는 신뢰 불가.
- 벤더 주장("10배 빠름")과 독립 검증을 구분 표기한다.

### 5. 불확실성 표시
못 찾거나 출처가 갈리면 `UNVERIFIED`/`CONFLICTING` 섹션에 적고 SME 판단에 넘긴다. 상충 시 한쪽을 임의로 고르지 않고 양쪽을 병기한다(출처 병기 원칙).

### 6. `_workspace/01_research.md` 작성
에이전트 정의의 출력 골격(claims 표 / 버전·현행성 / 벤치마크 / UNVERIFIED / 추가 발견)을 채운다.

## 산출물 기준
- 모든 사실 주장에 출처 URL + 날짜 + 확신도
- 벤치마크엔 조건(환경·규모·날짜·독립/벤더) 포함
- 확인 못 한 항목이 UNVERIFIED로 분리됨(빈 곳을 추측으로 채우지 않음)
- brief 요청 범위를 벗어나지 않음

## 후속/부분 재실행
`01_research.md`가 있으면 표에 행만 추가하고 UNVERIFIED 항목 재확인을 우선한다. SME가 작성 중 추가 검증을 요청하면 해당 항목만 보강한다.
