# TPO Brief: TOP 버튼 → 스마트 플로팅 액션 위젯 (Reading Assistant Widget)

## 기본 정보
- 요청 유형: UI 개선 + 기능 확장 (기획 → 디자인 → 구현 풀 파이프라인)
- 산출물: 블로그 독자용 플로팅 보조 위젯 (포스트가 아닌 기능 개발 건)
- 대상 사용자: DGPR 기술 블로그 독자 — **주로 미들~시니어 엔지니어**가 긴 기술 글(시스템 설계, Java, LLM, OS 등)을 정독하는 시나리오
- 수정 대상 파일: `js/top.js`, `_sass/_base.scss`, `_includes/footer.html`
- 제약: Jekyll 정적 사이트 / Vanilla JS(+jQuery) / CSS custom properties 다크·라이트 / `prefers-reduced-motion` 존중 / 모바일 반응형 / 기존 `#topBtn` ID 유지

## 1. 사용자 페인포인트 분석

긴 기술 글을 정독하는 독자가 현재 겪는 네비게이션 불편:

| # | 페인포인트 | 현재 상태 | 영향 |
|---|-----------|----------|------|
| P1 | **현재 위치 감각 상실** — 글이 얼마나 남았는지 알 수 없음 | 스크롤바만 존재, `scrolledPercentage` 계산은 하지만 UI에 미반영 | 완독률 저하, 이탈 |
| P2 | **TOP 버튼이 80%에서야 등장** — 글 중반에 위로 가고 싶어도 버튼 없음 | 80% 임계값 고정 | 핵심 기능이 늦게 노출 |
| P3 | **섹션 간 점프 불가** — 긴 글에서 특정 H2로 이동하려면 수동 스크롤 | 데스크탑 TOC(`wiki.html`/`post.html`)는 있으나 모바일/스크롤 중에는 접근성 낮음 | 탐색 비용 증가 |
| P4 | **좋은 글을 공유할 수단이 글 안에 없음** | 별도 공유 UI 부재 | 유입·확산 기회 손실 |
| P5 | **단일 텍스트 "Top" 버튼의 낮은 시인성/정보량** | 원형 버튼에 "Top" 텍스트만 | 브랜드·완성도 부족 |

## 2. 기능 정의 및 우선순위

단순 TOP을 넘어선 후보 기능을 **이 블로그 환경 적합성** 기준으로 선별.
정적 사이트 + Vanilla JS로 무리 없이 구현 가능한 범위로 한정한다.

### P0 (필수 — 이번 구현 핵심)
1. **스크롤 진행률 시각화 (Reading Progress Ring)**
   - 원형 버튼 테두리를 progress ring으로 활용해 읽은 비율(`scrolledPercentage`) 표시
   - 기존 계산 로직 재사용 → 추가 비용 최소. P1 직접 해결
2. **맨 위로 이동 (Top) — 등장 임계값 조정**
   - 80% → **약 20~25%(또는 1 viewport 스크롤 시)** 로 완화해 조기 노출. P2 해결
   - 진행률 ring 중앙에 위쪽 화살표 아이콘 배치 (텍스트 "Top" 제거). P5 해결

### P1 (권장 — 위젯 확장)
3. **섹션 빠른 이동 (Quick Section Nav)**
   - 위젯을 hover/클릭 시 펼쳐지는 액션 그룹으로 확장, 글 내 H2 목록으로 점프
   - **검증 필요**: 포스트 본문 H2에 안정적 `id`가 부여되는지(kramdown `auto_ids` 기본 on이나 실제 렌더 확인 요). 없으면 JS에서 동적 id 생성으로 대응
   - 데스크탑 TOC와 **중복되지 않도록** — 모바일/태블릿 우선, 데스크탑은 TOP+진행률만 노출하는 분기 고려
4. **현재 글 공유 (Share)**
   - 펼침 액션에 공유 버튼: `navigator.share` 지원 시 네이티브 공유 시트, 미지원 시 URL 클립보드 복사 fallback. P4 해결

### P2 (보류 — 이번 범위 제외, 향후 검토)
- 다크/라이트 토글 중복 배치(헤더에 이미 `theme-toggle.js` 존재 → 제외)
- 댓글로 점프(giscus) — 가치 대비 복잡도 낮은 우선순위
- 폰트 크기 조절 — 범위 초과

**권장 MVP 범위**: P0(1,2) + P1(3 또는 4 중 하나). 디자이너가 펼침 인터랙션 복잡도를 평가해 P1 포함 여부 최종 결정.

## 3. UX 원칙 — "콘텐츠가 주인공"

1. **비방해성(Non-intrusive)**: 평상시 단일 원형 버튼 크기 유지(현 50px 기준). 본문을 가리지 않음. 액션은 사용자가 의도적으로 펼칠 때만 노출
2. **점진적 공개(Progressive disclosure)**: 기본=진행률+TOP / 확장=섹션·공유. 정보 밀도를 단계적으로
3. **즉각적 피드백**: progress ring은 스크롤에 실시간 반응하되 `requestAnimationFrame`/throttle로 성능 확보 (현재 raw `scroll` 리스너 개선 필요)
4. **접근성**: 버튼에 `aria-label`, 키보드 포커스 가능, `prefers-reduced-motion` 시 ring 애니메이션·smooth scroll 비활성화. WCAG AA 대비
5. **테마 일관성**: progress는 `--color-interactive`/`--color-accent`, 트랙은 `--color-border`, 표면은 `--color-surface` 사용. 하드코딩 색상 금지
6. **모바일 우선**: 터치 타깃 최소 44px, 우하단 safe-area 고려, 펼침 방향은 위쪽

## 4. 성공 기준 (구현 후 개선 판정)

| 영역 | 기준 |
|------|------|
| 기능 | progress ring이 스크롤 비율과 정확히 동기화(0~100%), TOP 클릭 시 상단 이동 동작 |
| 노출 시점 | 첫 1 viewport 스크롤 이후 위젯 노출 (80% 대기 문제 해소) |
| 성능 | 스크롤 핸들러가 메인 스레드 블로킹 없이 60fps 유지(throttle/rAF 적용), Lighthouse 성능 회귀 없음 |
| 접근성 | `aria-label` 존재, 키보드 조작 가능, `prefers-reduced-motion` 시 애니메이션 정지, 색 대비 AA 충족 |
| 호환성 | 기존 `#topBtn` ID 유지로 회귀 없음, 다크/라이트 양 모드 정상 렌더, 375px 모바일 정상 |
| 비방해성 | 데스크탑 기존 TOC와 기능 중복/충돌 없음, 본문 가독 영역 침범 없음 |

## 5. 기술 검증 포인트 (→ Frontend 확인 요청)
- **검증 필요**: 포스트 본문 H2/H3에 kramdown `auto_ids`로 앵커 id가 실제 부여되는지 (P1-3 섹션 점프 전제)
- **검증 필요**: 데스크탑 레이아웃(`wiki.html`, `post.html`)의 기존 TOC와 위젯 섹션 네비 기능 중복 → 데스크탑/모바일 분기 노출 전략 확정
- **검증 필요**: `navigator.share` 브라우저 지원 범위 및 클립보드 fallback UX
- **검증 필요**: 현재 raw `scroll` 리스너를 rAF/throttle로 교체 시 `scrolledPercentage` 정확도 유지
