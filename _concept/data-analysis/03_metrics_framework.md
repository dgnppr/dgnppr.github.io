"어떤 지표가 좋아졌을 때 우리가 잘하고 있다고 말할 수 있는가"가 명확하지 않으면, 리텐션 분석이나 A/B 테스트 같은 개별 기법의 결과를 일관되게 해석할 수 없다. 지표 체계는 분석 기법보다 먼저 설계해야 한다.

---

## 지표 없이 분석하면 생기는 일

지표 체계가 없으면 팀마다 다른 숫자를 보고 다른 결론을 낸다. 마케팅팀은 신규 가입자가 늘었다고 좋다 하고, 제품팀은 리텐션이 떨어졌다고 나쁘다 한다. 둘 다 맞는 말이지만 회사 전체가 잘하고 있는지 못하고 있는지 알 수 없다.

지표 체계는 이 혼란에 구조를 부여한다. "어떤 하나의 숫자가 우리가 고객에게 가치를 만들고 있음을 보여주는가"부터 시작한다.

---

## 지표 계층 구조

```
NSM (North Star Metric)
  └─ L1: Primary Metrics (팀/제품 단위)
       └─ L2: Secondary Metrics (L1을 설명하는 하위 지표)

Guardrail Metrics (어떤 레벨에도 독립적으로 존재)
```

| 레벨 | 개수 | 역할 |
|---|---|---|
| **NSM** | 1개 | 회사 전체 방향 정렬 |
| **L1** | 팀당 3~5개 | 팀이 직접 움직일 수 있는 지표 |
| **L2** | L1당 2~4개 | L1 변화의 원인 진단 |
| **Guardrail** | 전사 3~5개 | 최적화 중 망가지면 안 되는 지표 |

---

## North Star Metric

NSM은 회사가 **고객에게 제공하는 핵심 가치**를 숫자로 표현한 것이다. 매출이 아니라 가치다. 매출은 결과고, NSM은 그 결과를 만드는 원인이어야 한다.

### 좋은 NSM의 5가지 조건

1. **고객 가치를 반영한다** — "이 숫자가 높으면 사용자가 제품에서 진짜 가치를 얻고 있다"고 말할 수 있어야 한다
2. **비즈니스 성과와 연결된다** — NSM이 오르면 장기적으로 매출도 따라온다는 인과가 있어야 한다
3. **팀이 직접 영향을 줄 수 있다** — 경쟁사 행동이나 시장 상황이 아니라 우리 행동이 움직일 수 있어야 한다
4. **측정 가능하다** — 매주 또는 매일 확인할 수 있어야 한다
5. **단일 숫자다** — 두 개의 NSM은 없다. 선택을 강제하는 게 NSM의 역할이다

### 업종별 NSM 예시

| 회사 | NSM | 의미 |
|---|---|---|
| Airbnb | 예약 숙박 일수 | 호스트와 게스트 모두에게 가치가 발생한 순간 |
| Slack | 팀당 주간 메시지 수 | 팀이 실제로 소통하고 있다는 신호 |
| Spotify | 월간 청취 시간 | 음악을 즐기는 핵심 가치 |
| Netflix | 주간 시청 시간 | 구독 갱신의 선행 지표 |
| 쿠팡 | 주문 완료 건수 (로켓배송) | 핵심 서비스의 활용 빈도 |

### NSM 설계 함정: 매출을 NSM으로 쓰지 않는 이유

매출은 훌륭한 후행 지표다. 하지만 NSM으로 쓰면 팀이 단기 매출에 집중한다. Airbnb가 "총 거래액"을 NSM으로 쓰면 고가 숙소에만 집중하고 저가 숙소 호스트를 소홀히 한다. "숙박 일수"는 모든 가격대의 예약을 동등하게 센다.

### 실제 NSM 선정 과정 — 국내 음식 구독 스타트업

**핵심 가치 정의**: "바쁜 직장인이 매일 건강하고 맛있는 식사를 준비하지 않아도 된다"

| 후보 | 고객 가치 | 비즈니스 연결 | 팀 영향 | 측정 | 탈락 이유 |
|---|---|---|---|---|---|
| 월간 앱 DAU | △ | △ | ✓ | ✓ | 앱을 열기만 해도 카운트 — 가치 없는 방문 포함 |
| 월간 구독 매출 | ✗ | ✓ | △ | ✓ | 가격 인상만으로도 오름 — 고객 가치 미반영 |
| 구독 갱신율 | ✓ | ✓ | △ | ✓ | 월 1회 측정 — 팀 의사결정에 너무 느림 |
| **월간 식사 배달 완료 건수** | **✓** | **✓** | **✓** | **✓** | **선정** |

"월간 식사 배달 완료 건수"를 선정한 이유:
- 배달 완료 = 고객이 실제로 가치를 받은 순간 (앱을 열었거나 구독한 것과 다름)
- 이 숫자가 오르면 구독 갱신율, 매출이 따라온다는 6개월 상관 데이터가 존재했음
- 메뉴 품질, 배달 시간, 온보딩 개선이 모두 이 숫자에 직접 연결됨

## Guardrail 지표

Guardrail은 **최적화 도중 망가지면 안 되는 지표**다. A/B 테스트에서 Primary Metric은 올랐지만 Guardrail이 나빠졌다면 그 실험은 성공이 아니다.

### 왜 필요한가

```
실험 목표: 홈 화면 CTR 개선
결과: CTR +8% (성공처럼 보임)
       ↓
하지만 Guardrail을 보면:
  - 구매 전환율 -3%   ← Guardrail 위반
  - 세션 이탈률 +5%   ← Guardrail 위반
```

클릭만 유도하는 낚시성 배너가 CTR은 올리지만 실제 가치는 훼손했다.

### 실제 Guardrail 위반 사례

**실험**: 홈 화면 배너 개수를 3개 → 5개로 늘리는 A/B 테스트 (커머스 앱)

| 지표 | Control | Treatment | 변화 | 판정 |
|---|---|---|---|---|
| 배너 CTR (Primary) | 4.2% | 5.1% | **+21%** | ✅ 개선 |
| 첫 화면 로드 시간 | 1.2s | 2.0s | **+800ms** | ❌ Guardrail 위반 |
| 장바구니 전환율 | 8.7% | 8.2% | **-5.7%** | ❌ Guardrail 위반 |
| Day 7 리텐션 | 41% | 39% | **-2%** | ❌ Guardrail 위반 |

결론: **실험 중단**. CTR만 보면 성공이지만 페이지가 느려지고 실제 구매가 줄었다. Guardrail 없이 출시했다면 단기 CTR 보고서는 성공으로 기록되고 리텐션 하락의 원인은 한참 뒤에야 발견됐을 것이다.

### 전사 공통 Guardrail 예시

| Guardrail 지표 | 방어하는 것 |
|---|---|
| 월간 활성 사용자 이탈률 | 성장 지표 개선이 기존 사용자 손해로 이루어지는 것 |
| 고객 서비스 문의율 | 기능 변경이 사용자 혼란을 유발하는 것 |
| 페이지 로드 시간 | 기능 추가가 성능 저하로 이어지는 것 |
| 결제 에러율 | 전환율 실험이 결제 안정성을 훼손하는 것 |

### Primary Metric vs Guardrail의 차이

| | Primary Metric | Guardrail Metric |
|---|---|---|
| 역할 | 개선 목표 | 최소 기준 유지 |
| 팀 책임 | 올려야 한다 | 내리면 안 된다 |
| A/B 테스트에서 | 개선 여부 판단 | 이 기준 위반 시 실험 무효 |

## Leading vs Lagging 지표

| | Leading 지표 | Lagging 지표 |
|---|---|---|
| 정의 | 미래 결과를 예측하는 선행 신호 | 이미 발생한 결과 |
| 반응 속도 | 빠름 (수일~수주) | 느림 (수주~수개월) |
| 개입 가능성 | 높음 | 낮음 (결과는 바꿀 수 없음) |
| 예시 | 신규 사용자의 첫 7일 핵심 행동 완료율 | 30일 리텐션, 월간 매출 |

### 실전 적용

리텐션이 문제라면, 리텐션 자체(lagging)를 보는 것만으로는 느리다. 리텐션과 강한 상관을 갖는 초기 행동(leading)을 찾아야 한다.

```
Lagging: Day 30 리텐션
  ↑
Leading: 가입 후 7일 이내 핵심 기능 사용 여부
  ↑
Leading의 Leading: 온보딩 완료율
```

이 구조를 만들면 리텐션이 나빠지기 전에 온보딩 완료율이 먼저 신호를 보낸다.

### Magic Number 찾기

"이 행동을 완료한 사용자는 30일 후에도 남아있다"는 임계값을 찾는 것이 핵심 leading 지표 발굴 방법이다.

```python
import pandas as pd
from scipy import stats

# 예: 첫 7일 내 '핵심 기능 사용 횟수'와 Day 30 리텐션의 관계
df = events_df.groupby('user_id').agg(
    core_actions_7d=('event_name', lambda x: (x == 'core_feature_used').sum()),
    retained_day30=('retained_day30', 'first')
)

# 임계값별 리텐션율 비교
for threshold in range(0, 10):
    group_a = df[df['core_actions_7d'] >= threshold]['retained_day30'].mean()
    group_b = df[df['core_actions_7d'] < threshold]['retained_day30'].mean()
    print(f"≥{threshold}회: {group_a:.1%} vs <{threshold}회: {group_b:.1%}")
```

**실제 분석 결과 예시** — 배달 앱, 첫 7일 주문 횟수와 Day 30 리텐션:

| 첫 7일 주문 횟수 | Day 30 리텐션 | 사용자 수 |
|---|---|---|
| 0회 | 6% | 12,400명 |
| 1회 | 28% | 8,900명 |
| **2회** | **51%** | 4,200명 |
| 3회 | 63% | 2,100명 |
| 4회 이상 | 71% | 1,800명 |

1회 → 2회 구간에서 28% → 51%로 가장 큰 폭으로 뛴다. **"첫 7일 내 2회 주문"이 Magic Number**.

이 발견이 지표 체계로 연결되면:
- **L2 leading 지표**: "신규 가입자의 첫 7일 2회 주문 달성률" (현재 32%)
- **온보딩팀 목표**: 이 비율을 40%로 올리기
- **A/B 테스트 Primary Metric**: 위 지표 — Day 30 리텐션을 기다리지 않고 2주 안에 실험 결과를 볼 수 있다

## Goodhart's Law

> "지표가 목표가 되는 순간, 그 지표는 좋은 지표가 아니게 된다."

지표를 KPI로 만들면 팀은 그 숫자를 올리는 데 집중하다 본질을 잃는다.

### 실사례

| KPI | 왜곡 |
|---|---|
| 콘텐츠 게시글 수 | 저품질 짧은 글을 대량 생산 |
| 고객 서비스 처리 건수 | 문제를 해결하지 않고 티켓을 빠르게 닫음 |
| 앱 리뷰 평점 | 앱 내에서 좋은 시점에 리뷰 유도 팝업 반복 |
| Day 1 리텐션 | 푸시 알림 남발로 단기 재방문 유도, 장기 이탈 가속 |

### 대응 방법

1. **지표 쌍으로 관리**: 하나의 지표를 올릴 때 반드시 견제하는 짝 지표를 둔다. 콘텐츠 게시글 수 → 게시글 수 + 인당 조회수 쌍으로 본다.
2. **지표를 정기적으로 교체**: 특정 지표가 최적화됐다 싶으면 새로운 leading 지표로 이동한다.
3. **지표 설계를 공개**: 어떤 지표를 왜 쓰는지 팀 전체가 알면 왜곡이 줄어든다.

## 지표 체계 구축 프로세스

### Step 1: 핵심 고객 가치 정의

"우리 제품이 없으면 고객이 무엇을 못 하는가?" 한 문장으로 쓴다.

### Step 2: NSM 후보 3개 도출, 기준으로 평가

| 후보 | 고객 가치 반영 | 비즈니스 연결 | 팀 영향 가능 | 측정 가능 |
|---|---|---|---|---|
| 주문 완료 건수 | ✓ | ✓ | ✓ | ✓ |
| 총 거래액 | △ | ✓ | △ | ✓ |
| 앱 세션 수 | △ | △ | ✓ | ✓ |

위 기준으로 평가해 하나를 고른다.

### Step 3: L1 지표 정의

각 팀이 NSM에 기여하는 방식을 지표로 표현한다. 팀당 5개를 넘기면 집중도가 떨어진다.

```
NSM: 주문 완료 건수
  ├─ 성장팀 L1: 신규 가입자 수, 첫 구매 전환율
  ├─ 제품팀 L1: 검색 클릭률, 장바구니 전환율
  └─ 리텐션팀 L1: Day 30 리텐션, 재구매율
```

### 실제 지표 체계 예시 — 커머스 앱

**핵심 가치**: "원하는 물건을 빠르고 믿을 수 있게 받는다"

```
NSM: 월간 주문 완료 건수

Guardrail (전사):
  - 결제 에러율 < 0.3%
  - 앱 크래시율 < 0.1%
  - CS 문의율 < 주문 대비 2%
  - 앱 첫 화면 로드 시간 < 1.5s

성장팀
  L1: 신규 가입자 수, 신규 가입자의 첫 주문 전환율
  L2: 채널별 CPI, 랜딩→회원가입 전환율, 회원가입→첫 주문 소요 일수

제품팀
  L1: 검색 클릭률, 장바구니→주문 전환율
  L2: 검색 결과 0건 비율, 상품 상세→장바구니 전환율, 장바구니 이탈 원인 Top 3

리텐션팀
  L1: Day 30 리텐션, 월간 재구매율
  L2: 첫 7일 2회 주문 달성률(Magic Number), 두 번째 주문까지 소요 일수
```

각 팀은 자신의 L1을 올리는 데 집중하되, Guardrail이 위반되면 전 팀이 해당 지표 복구를 다음 스프린트 1순위로 올린다.

### Step 4: Guardrail 지표 정의

각 팀의 L1 최적화가 다른 팀의 영역이나 사용자 경험 전반을 훼손하지 않도록 전사 수준의 Guardrail을 정의한다.

### Step 5: 대시보드 구성

- **전사 대시보드**: NSM + Guardrail, 주간 단위 추이
- **팀 대시보드**: L1 + L2, 일간 단위 추이, 실험 결과 포함

## 전체 체크리스트

**NSM**
- 고객 가치를 반영하는가? (매출이 아닌가?)
- 팀이 직접 영향을 줄 수 있는가?
- 단일 지표인가?

**Guardrail**
- A/B 테스트마다 Guardrail을 확인하는 프로세스가 있는가?
- Guardrail이 위반됐을 때 실험을 중단하는 기준이 있는가?

**Leading/Lagging**
- 각 L1 지표의 leading 지표가 정의됐는가?
- Magic Number를 데이터로 검증했는가?

**Goodhart's Law 방어**
- 모든 KPI에 견제 지표(paired metric)가 있는가?
- 지표 선정 이유와 방법론이 팀에 공개됐는가?
