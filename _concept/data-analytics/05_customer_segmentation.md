세그먼트는 사용자를 나누는 라벨이 아니라 **분석과 액션의 단위**다. 세그먼트 정의가 틀리면 그 위에 얹은 모든 지표 — 세그먼트별 전환율, LTV, 리텐션 — 가 함께 무의미해진다. Funnel이 수학적으로 성립하기 위한 필요조건이 있듯, 세그먼트도 성립 조건이 있다. 이 글은 세그먼트를 "정의하는 축", "성립 조건", "만드는 방법", "검증 방법" 순으로 다룬다.

RFM 세그멘테이션 자체는 [행동 데이터 분석 핵심 기법 5가지](/concept/data-analytics/01_behavioral_analytics_techniques)에서 다뤘다. 여기서는 RFM을 포함한 모든 세그먼트에 공통으로 적용되는 정의·검증 원칙에 집중한다.

## 1. 세그먼트를 정의하는 축

세그먼트는 어떤 기준으로 나누느냐에 따라 답하는 질문이 다르다. 기준을 먼저 정하지 않고 나누면, 나온 그룹이 무엇을 의미하는지 사후에 끼워맞추게 된다.

| 축 | 정의 predicate 예시 | 답하는 질문 | 안정성 |
|---|---|---|---|
| **속성 기반** (demographic/firmographic) | 연령대, 지역, 가입 채널, 기업 규모 | 누가 우리 고객인가 | 높음 (거의 안 바뀜) |
| **행동 기반** (behavioral) | 최근 30일 로그인 횟수, 사용 기능 조합 | 어떻게 쓰는가 | 낮음 (자주 바뀜) |
| **가치 기반** (value) | RFM, 누적 매출, 예측 LTV | 얼마나 중요한가 | 중간 |
| **생애주기** (lifecycle) | 신규 / 활성 / 휴면 / 이탈 | 지금 어느 단계인가 | 중간 (단계는 이동) |
| **니즈·태도** (needs) | 설문·리서치 기반 페르소나 | 왜 쓰는가 | 높음이나 측정 어려움 |

두 가지가 실무에서 자주 엉킨다.

- **속성 축과 행동 축을 한 세그먼트에 섞는 것**: "20대 헤비유저"는 두 축의 교차다. 교차는 가능하지만, 교차하는 순간 각 셀의 크기가 급감하고(1번 축 × 2번 축) 통계적 검정력이 무너진다. 교차는 의도적으로, 크기를 확인하고 써야 한다.
- **니즈 축을 행동 데이터로 추정할 수 있다는 착각**: "이 기능을 쓰니까 이런 니즈일 것"은 가설이지 사실이 아니다. 니즈 축은 리서치로만 확정된다.

> **규칙**: 세그먼트를 만들기 전에 "이건 어느 축의 세그먼트인가"를 한 문장으로 정의한다. 축을 못 정하면 그건 세그먼트가 아니라 필터다.

## 2. 성립하는 세그먼트의 6가지 조건

세그먼트 체계가 분석·액션의 단위로 쓰이려면 아래 6가지를 만족해야 한다. 하나라도 어기면 세그먼트별 비교가 깨진다.

### 2.1 완전 분할 또는 명시적 중첩 정책 (MECE)

한 사용자가 어느 세그먼트에 속하는지가 유일하게 결정되거나 (mutually exclusive), 중첩을 허용한다면 그 정책을 명시해야 한다.

중첩을 방치하면 세그먼트별 합계가 전체를 초과하고, "세그먼트 A의 매출 기여 40%, B가 35% ..."의 합이 100%를 넘는 모순이 생긴다.

```
-- 배타형(가장 흔함): 각 사용자를 한 세그먼트에만 배정
CASE
  WHEN last_active_days <= 7  THEN 'active'
  WHEN last_active_days <= 30 THEN 'dormant'
  ELSE 'churned'
END AS lifecycle_segment  -- 조건이 서로 겹치지 않고, ELSE로 전체를 덮음
```

> **규칙**: 배타형이면 조건 구간이 겹치지 않고 ELSE가 나머지를 전부 덮는지 확인한다. 중첩형이면 "한 사용자가 여러 세그먼트에 속할 수 있음"을 리포트에 명시하고, 합계 지표를 쓰지 않는다.

### 2.2 안정성 (Stability)

세그먼트 멤버십이 관측 노이즈로 매일 바뀌면 안 된다. 어제 'Champion'이던 사람이 오늘 'At Risk'가 되는 일이 잦으면, 그 세그먼트를 대상으로 한 캠페인은 대상을 조준할 수 없다.

안정성은 **전이 행렬(transition matrix)**로 측정한다 (5절). 임계는 도메인마다 다르지만, 경험적으로 한 기간에서 다음 기간까지 같은 세그먼트에 남는 비율(대각 성분)이 60% 미만이면 세그먼트 경계나 측정 윈도우를 의심해야 한다.

### 2.3 액션 가능성 (Actionability)

세그먼트마다 **다른 행동을 유발**해야 한다. 두 세그먼트에 똑같은 조치를 한다면 그 둘은 나눌 이유가 없다.

"고가치 vs 저가치"로 나눴는데 양쪽에 같은 프로모션을 보낸다면, 그 분할은 리포트용 장식일 뿐이다.

### 2.4 충분한 크기 (Size)

각 세그먼트가 통계적 검정력을 가질 만큼 커야 한다. 세그먼트별 전환율을 비교하려면 각 셀에 최소 표본이 있어야 한다.

비율 지표의 표준오차는 대략

$$SE = \sqrt{\frac{p(1-p)}{n}}$$

전환율 $p \approx 0.05$인 세그먼트에서 오차를 ±1%p로 잡으려면 $n \approx 1{,}800$이 필요하다. 셀이 이보다 작으면 "세그먼트 A가 B보다 전환율 높다"는 노이즈일 수 있다.

> **규칙**: 세그먼트를 쪼갤 때마다 각 셀의 n을 확인한다. 축을 교차할수록 셀은 곱셈으로 작아진다.

### 2.5 분리도 (Separation)

세그먼트들이 관심 결과 지표(리텐션, LTV, 전환율)에서 **실제로 달라야** 한다. 입력 피처는 다른데 결과가 같으면, 비용을 들여 나눌 이유가 없다.

분리도는 세그먼트를 만든 피처가 아니라 **만드는 데 쓰지 않은 결과 지표**로 검증해야 한다 (4.2). RFM으로 나눴으면 RFM이 아닌 '다음 분기 재구매율'로 분리도를 본다.

### 2.6 측정 가능한 predicate

세그먼트 정의는 데이터에서 재현 가능하게 계산되어야 한다. "충성 고객"은 세그먼트 이름이지 정의가 아니다. `최근 90일 3회 이상 구매 AND 누적 매출 상위 20%`가 정의다.

> **규칙**: 세그먼트 이름 옆에 항상 실행 가능한 predicate를 붙인다. 이름만 있고 predicate가 없으면 사람마다 다른 집단을 같은 이름으로 부른다.

## 3. 세그먼트를 만드는 두 가지 방법

세그먼트 생성은 크게 규칙 기반(rule-based)과 비지도 군집(unsupervised clustering) 둘로 나뉜다.

| 방법 | 정의 방식 | 장점 | 단점 | 언제 |
|---|---|---|---|---|
| **규칙 기반** | 사람이 임계값·조건을 지정 (RFM, 생애주기) | 해석 명확, 액션 직결, 재현 쉬움 | 사람의 가정에 의존, 상호작용 놓침 | 축과 임계가 명확할 때 (대부분의 경우) |
| **군집** | 알고리즘이 데이터에서 그룹 발견 (K-means 등) | 예상 못 한 패턴 발견, 다차원 상호작용 반영 | 해석 어려움, 스케일·K에 민감, 불안정 | 축을 모르고 탐색할 때 |

실무 기본값은 규칙 기반이다. 규칙 기반이 답을 못 줄 때, 즉 "어떤 축으로 나눠야 할지 모르겠다"일 때만 군집으로 간다.

### 3.1 규칙 기반

RFM이 대표 사례이며 [기존 글](/concept/data-analytics/01_behavioral_analytics_techniques)에서 SQL과 함께 다뤘다. 규칙 기반의 핵심은 임계값을 **분포를 보고** 정하는 것이다. `NTILE(5)` 같은 분위수는 분포를 균등하게 자를 뿐, 그 경계에 실제 행동 차이가 있는지는 별개다 (2.5 분리도로 검증).

### 3.2 K-means 군집

축을 모를 때, 여러 피처를 동시에 넣고 자연스러운 그룹을 찾는다. 세 가지를 반드시 지켜야 한다.

**(1) 피처 스케일링 필수.** K-means는 유클리드 거리를 쓴다. `누적매출(0~1,000,000)`과 `방문횟수(0~50)`를 그대로 넣으면 매출이 거리를 지배해 방문횟수는 무시된다. 표준화(z-score)나 로그 변환이 선행되어야 한다.

**(2) 긴 꼬리 분포는 로그 변환.** 매출·방문 같은 지표는 우측으로 심하게 치우친다. 로그를 취하지 않으면 소수 극단값이 군집을 지배한다.

**(3) K는 실루엣으로 고른다.** 엘보우는 주관적이다. 실루엣 계수를 병행한다.

```python
import numpy as np
import pandas as pd
from sklearn.preprocessing import StandardScaler
from sklearn.cluster import KMeans
from sklearn.metrics import silhouette_score

# 예시 데이터: user_id별 집계 피처
df = pd.DataFrame({
    'user_id':   [1, 2, 3, 4, 5, 6],
    'recency':   [3, 40, 5, 90, 2, 60],       # 마지막 구매 경과일
    'frequency': [12, 2, 9, 1, 15, 3],        # 구매 횟수
    'monetary':  [820_000, 40_000, 610_000, 15_000, 970_000, 55_000],
})

# (2) 긴 꼬리 피처는 로그, (1) 그 뒤 표준화
X = df[['recency', 'frequency', 'monetary']].copy()
X['monetary'] = np.log1p(X['monetary'])
X['frequency'] = np.log1p(X['frequency'])
X_scaled = StandardScaler().fit_transform(X)

# (3) K를 실루엣으로 선택
for k in range(2, 6):
    labels = KMeans(n_clusters=k, n_init=10, random_state=42).fit_predict(X_scaled)
    print(k, round(silhouette_score(X_scaled, labels), 3))
```

실루엣 계수는 각 점이 자기 군집에 얼마나 잘 맞고 옆 군집과 얼마나 떨어졌는지를 $[-1, 1]$로 준다.

$$s(i) = \frac{b(i) - a(i)}{\max\{a(i),\, b(i)\}}$$

$a(i)$는 같은 군집 내 평균 거리, $b(i)$는 가장 가까운 다른 군집까지의 평균 거리다. 전체 평균이 0.5 이상이면 구조가 뚜렷한 편, 0.25 이하면 군집이 인위적일 가능성이 크다.

> **한계**: K-means는 구형·등분산 군집을 가정한다. 밀도가 다르거나 비구형 군집(예: 계정 유형이 섞인 데이터)에는 맞지 않는다. 그럴 땐 GMM이나 DBSCAN을 고려하되, 해석 비용이 더 든다. 대부분의 고객 세그먼트는 K-means로 충분하고, 안 되면 규칙 기반으로 후퇴하는 편이 낫다.

## 4. 세그먼트 검증 방법

세그먼트를 만들었으면 쓰기 전에 검증한다. 검증 없이 배포된 세그먼트는 대개 크기가 극단적으로 쏠려 있거나 결과가 분리되지 않는다.

### 4.1 크기 분포 점검

가장 먼저, 세그먼트별 사용자 수를 본다. 한 세그먼트가 90%를 차지하면 나머지는 노이즈이고, 균등하게 6등분되면 인위적으로 자른 것(NTILE 부작용)일 수 있다.

```
SELECT segment, COUNT(*) AS n,
       ROUND(COUNT(*) / SUM(COUNT(*)) OVER (), 3) AS share
FROM user_segments
GROUP BY segment
ORDER BY n DESC
```

### 4.2 분리도 검정 — 만들 때 안 쓴 지표로

세그먼트를 만든 피처가 아닌 **결과 지표**에서 세그먼트가 갈리는지 본다. 이게 세그먼트가 실제로 유용한지의 핵심 증거다.

```
-- RFM 세그먼트가 '다음 분기 재구매율'을 실제로 가르는가
SELECT s.segment,
       COUNT(*)                                  AS users,
       AVG(CASE WHEN n.repurchased THEN 1 ELSE 0 END) AS repurchase_rate
FROM user_segments s
LEFT JOIN next_quarter_repurchase n USING (user_id)
GROUP BY s.segment
ORDER BY repurchase_rate DESC
```

세그먼트 간 재구매율이 거의 같다면(예: 전부 12~14%), RFM 분할은 이 결과 지표에 대해 아무 정보가 없다. 세그먼트를 다시 정의하거나, 다른 축을 써야 한다.

### 4.3 안정성 검정

5절의 전이 행렬로 멤버십이 기간 간에 유지되는지 본다.

## 5. 세그먼트 안정성 — 전이 행렬

두 시점(예: 이번 달 vs 지난 달)에 각 사용자가 어느 세그먼트에 있었는지를 교차표로 만든다. 대각선(같은 세그먼트 유지)이 두꺼울수록 안정적이다.

```
WITH m1 AS (  -- 지난달 세그먼트
  SELECT user_id, segment AS seg_prev FROM user_segments_2026_05
),
m2 AS (       -- 이번달 세그먼트
  SELECT user_id, segment AS seg_curr FROM user_segments_2026_06
)
SELECT seg_prev, seg_curr,
       COUNT(*) AS n,
       ROUND(COUNT(*) / SUM(COUNT(*)) OVER (PARTITION BY seg_prev), 3) AS row_share
FROM m1 JOIN m2 USING (user_id)
GROUP BY seg_prev, seg_curr
ORDER BY seg_prev, seg_curr
```

`row_share`의 대각 성분이 그 세그먼트의 유지율이다.

```
seg_prev   seg_curr    row_share
Champion   Champion    0.71   ← 안정 (유지율 71%)
Champion   Loyal       0.18
Champion   At Risk     0.11
At Risk    At Risk     0.42   ← 불안정 (경계 재검토 필요)
At Risk    Champion    0.20
At Risk    Lost        0.38
```

전이 행렬은 두 가지를 준다.

- **안정성 진단**: 대각 성분이 낮은 세그먼트는 경계가 노이즈에 걸쳐 있다.
- **생애주기 흐름**: 'At Risk → Lost'로 38%가 흘러간다면, At Risk 시점의 개입이 이탈 방지 지점이라는 신호다. (단, 이건 상관이다. 개입 효과는 A/B 테스트로 검증한다.)

## 6. 흔한 실수

1. **결과 지표로 세그먼트를 만들고 같은 지표로 검증**: LTV로 나눈 세그먼트가 LTV에서 다른 건 당연하다. 순환 논리다. 검증은 항상 만들 때 안 쓴 지표로 한다 (4.2).
2. **크기 확인 없이 축 교차**: "VIP × 신규 × 모바일"까지 쪼개면 셀이 수십 명으로 줄어 어떤 비교도 유의하지 않다.
3. **군집을 먼저, 해석을 나중에**: 축을 정하지 않고 K-means부터 돌리면, 나온 군집을 사후에 억지로 이름 붙이게 된다. 규칙 기반으로 답할 수 있으면 군집을 쓰지 않는다.
4. **세그먼트를 정적이라 가정**: 사용자는 세그먼트 사이를 이동한다. 스냅샷 한 장으로 캠페인을 설계하면 배포 시점엔 이미 대상이 바뀌어 있다.
5. **스케일링 없이 K-means**: 단위가 큰 피처가 거리를 독점한다 (3.2).

- 이 세그먼트는 어느 축인가 (속성/행동/가치/생애주기/니즈)를 한 문장으로 말할 수 있는가?

- 배타형인가 중첩형인가? 중첩이면 합계 지표를 피했는가?
- 각 세그먼트의 n이 결과 비교에 충분한가?
- 축을 교차했다면 각 셀 크기를 확인했는가?

**해석**
- 세그먼트마다 실제로 다른 액션이 나오는가?
