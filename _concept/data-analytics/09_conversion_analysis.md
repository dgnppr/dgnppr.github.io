유저가 의도한 목표 행동(구매, 가입, 구독)에 도달하는 비율을 분석하는 방법을 다룬다. 퍼널 설계의 기초는 [[00_how_to_design_funnel]], 실험 방법론은 [[10_trustworthy_controlled_experiments]]를 전제로 한다.

## 왜 분석하는가

전환 분석의 가치는 **지렛대의 위치를 알려준다**는 데 있다. 트래픽 2배는 예산 2배가 들지만, 전환율 2%→4%는 같은 트래픽으로 매출을 2배로 만든다. 그리고 전환율 개선의 재료는 언제나 퍼널의 병목 — 유저가 의도를 갖고 들어왔다가 새는 지점 — 에 있다.

이 분석이 직접 지탱하는 의사결정: **개선 투자의 우선순위**(어느 단계·세그먼트가 가장 비싼 병목인가), **실험 로드맵**(무엇을 테스트할 것인가), **캠페인·리타게팅 타이밍**(전환 지연 구조).

동시에 전환율은 AARRR 지표 중 **오독이 가장 쉬운 지표**다. 분모가 합의되지 않은 전환율, 유의성 없는 개선 보고, 선별 효과로 만든 착시 개선이 조직에 가장 자주 유통되는 잘못된 숫자들이다. 전환 분석의 절반은 계산이고 절반은 오독 방지다.

## 핵심 지표와 정의

| 지표 | 정의 시 결정할 것 |
|------|------------------|
| 전환율 | **분모가 무엇인가** — 세션 기준, 유저 기준, 노출 기준은 전부 다른 지표다. "전환율 3%"는 분모 없이는 숫자가 아니다 |
| 퍼널 단계 전환율 | 단계 간 이동의 판정 이벤트, 단계 순서 강제 여부(순서 무관 도달 vs 순차 통과) |
| 전환 소요 시간 | 첫 의도 신호부터 전환까지의 분포. 전환 윈도우 설정의 근거 |
| 마이크로 전환 | 최종 전환의 선행 행동(장바구니, 찜, 견적). 표본이 커서 실험 감도가 좋지만, 최종 전환과의 연결 검증 필요 |
| 전환수 vs 전환율 | 둘은 함께 움직이지 않는다. 진입 문턱을 높이면 전환율은 오르고 전환수는 준다 |

유저 기준 전환율은 기간 정의가 추가로 필요하다 — "그 주에 방문한 유저 중 그 주에 전환"(캘린더 기준)과 "그 주에 첫 방문한 유저 중 N일 내 전환"(코호트 기준)은 다른 지표이고, 후자가 개선 효과 측정에 적합하다. 캘린더 기준은 과거 유입의 지연 전환이 섞여 원인 귀속이 안 된다.

## 어떻게 분석하는가

### 1. 퍼널 분해 — 병목을 찾고, 세그먼트로 이질성을 확인한다

전체 퍼널의 예:

| 단계 | 도달 유저 | 단계 전환율 |
|------|----------|------------|
| 상품 상세 조회 | 100,000 | — |
| 장바구니 담기 | 18,000 | 18% |
| 결제 시작 | 9,000 | 50% |
| 결제 완료 | 5,400 | 60% |

전체 전환율 5.4%라는 숫자보다, "결제 시작→완료 60%"라는 단계 숫자가 액션으로 연결된다(결제 수단 실패율, 배송비 노출 시점 등 원인 후보가 좁혀지므로).

병목을 찾았으면 반드시 **세그먼트로 쪼갠다**: 플랫폼(웹/앱), 신규/재방문, 유입 채널, 디바이스. 전체 60%가 iOS 75% / Android 45%로 갈라지면 문제는 UX 일반론이 아니라 특정 플랫폼의 결함이다. 전환 개선의 실무는 "평균을 올리는 일"이 아니라 "평균을 끌어내리는 세그먼트를 찾는 일"에 가깝다.

### 2. 전환 지연 분석 — 윈도우를 데이터로 정한다

전환까지 걸리는 시간의 누적 분포를 그린다. 예: 전환의 60%가 당일, 85%가 7일 내, 95%가 30일 내. 이 분포가 결정하는 것들:

- **전환 윈도우**: 어트리뷰션·캠페인 평가의 윈도우를 관례(7일, 30일)가 아니라 자기 데이터의 분포로 정한다.
- **미성숙 코호트 처리**: 지난주 유입 코호트의 30일 전환율은 아직 존재하지 않는 숫자다. 미성숙 구간을 마스킹하거나 "N일 내 전환"으로 통일한다.
- **고관여 상품의 분석 단위**: 전환의 절반이 3회 이상 방문 후 일어난다면 세션 전환율은 구조적으로 낮게 나오는 지표이고, 유저 기준으로 봐야 한다.

### 3. 실험 — 전환율 개선 주장의 유일한 검증 수단

전환 개선은 상관 분석이 아니라 A/B 테스트로 검증한다. 실무에서 지켜야 할 최소선:

- **표본 크기를 먼저 계산한다.** 기준 전환율 5%에서 상대 10% 개선(5%→5.5%)을 검정력 80%, 유의수준 5%로 잡으려면 그룹당 약 3만 명이 필요하다. 이 계산 없이 시작한 실험은 "차이 없음"과 "표본 부족"을 구분하지 못한 채 끝난다.
- **peeking 금지.** 매일 p값을 보다가 유의해진 날 멈추면 1종 오류율이 설계값을 초과한다. 기간을 미리 정하거나 순차 검정 방법론을 쓴다.
- **가드레일 지표를 함께 본다.** 전환율이 오르면서 AOV·환불률·리텐션이 나빠지는 개선(강매형 UX)은 개선이 아니다.
- 상세한 방법론(SRM 체크, 다중 비교, 이질적 효과)은 [[10_trustworthy_controlled_experiments]] 참고.

### 4. 이탈 원인의 정성 결합

퍼널 데이터는 "어디서 새는가"까지만 말한다. "왜 새는가"는 이탈 지점의 세션 리플레이, 폼 필드 단위 이탈(어느 입력란에서 멈추는가), 이탈 직후 행동(검색어, 고객센터 문의), 설문으로 보강한다. 병목 발견(정량) → 원인 가설(정성) → 실험 검증(정량)이 전환 분석의 완전한 사이클이다.

## 분석용 마트를 어떻게 설계하는가

전환 분석의 반복 쿼리는 "매번 이벤트 원본에서 퍼널을 재조립"하는 방식으로는 지속 불가능하다. 구축 원칙은 [[2026-001-data-mart-checklist]]를 따른다.

| 마트 | 그레인 | 핵심 컬럼 | 용도 |
|------|--------|----------|------|
| 퍼널 단계 도달 팩트 | 유저(또는 세션) × 퍼널 × 단계 | 최초 도달 시각, 도달 세션 키 | 단계 전환율, 소요 시간 분포 |
| 전환 원장 | 전환 건 1행 | 전환 시각, 금액, transaction_id, 첫 의도 시각 | 전환수·지연 분석의 원천, dedup 기준 |
| 세션 마트 | 세션 1행 | 시작·종료, 유입 소스, 단계 최대 도달 | 세션 분모 전환율, 채널별 퍼널 |

설계에서 중요한 것들:

- **단계 판정 로직을 마트에 고정한다.** "결제 시작"이 어떤 이벤트(들)인지를 분석마다 다시 정하면 퍼널이 리포트마다 달라진다. 단계 → 이벤트 매핑은 마트의 변환 로직 한 곳에만 존재하고, 태깅 변경 시 이 매핑의 버전을 올린다.
- **wide 포맷이 실무에 유리하다.** 유저(세션) 1행에 단계별 최초 도달 시각을 컬럼으로 눕히면(`detail_at`, `cart_at`, `checkout_at`, `purchase_at`) 전환율·지연·중도 이탈이 전부 단순 비교 연산이 된다. 단계가 자주 바뀌는 초기에만 long 포맷을 유지한다.
- **분모 후보를 마트가 제공해야 한다.** 세션 기준·유저 기준·코호트 기준 분모가 전부 마트에서 바로 나와야 "분모 합의" 체크가 실제로 작동한다.
- **전환 원장에 dedup 키를 강제한다.** transaction_id 유니크 테스트가 없으면 재전송 중복이 전환율을 조용히 부풀린다.
- **실험 배정 테이블과 조인 가능하게 만든다.** 유저 키와 시간 기준이 실험 플랫폼의 배정 로그와 맞아야 실험 분석이 마트 위에서 돌아간다. 전환 마트 따로, 실험 분석 따로면 두 결과가 어긋나는 날이 온다.

## 다른 지표와 어떻게 엮어 읽는가

- **전환 × 획득**: 채널별 퍼널([[07_acquisition_analysis]]). 채널 간 전환율 차이는 UX가 아니라 유입 의도의 차이인 경우가 많다 — 검색 유입과 디스플레이 유입의 전환율을 같은 기준선으로 평가하면 안 된다. 반대로 같은 채널 안에서의 전환율 하락은 UX·가격 신호다.
- **전환 × 리텐션**: 전환율 개선이 만든 유저의 질을 확인하는 조합([[11_retention_analysis]]). 공격적 넛지·할인으로 만든 전환은 환불률과 이후 잔존에서 청구서가 온다. 실험 가드레일에 리텐션을 넣는 이유다.
- **전환 × 매출**: 전환율과 AOV의 트레이드오프([[10_revenue_analysis]]). 무료배송 문턱 인하가 전환율을 올리고 AOV를 내리는 식의 상충은 흔하며, 판정 지표는 단계 지표가 아니라 유저당 매출이다.
- **전환 × 인게이지먼트**: 활성화([[08_engagement_analysis]])와 결제 전환의 관계. 가치 경험 전에 결제를 요구하는 구조인지(전환 장벽), 가치 경험이 결제로 이어지지 않는 구조인지(수익화 설계)를 가르는 조합이다.
- **읽는 순서**: 전환율 변동 발견 → 분모·태깅 변경 확인(가짜 변동 소거) → 세그먼트·채널 분해(믹스인가 실질인가) → 단계 분해(어디인가) → 정성 조사(왜인가) → 실험(검증).

## 필수 체크리스트

- [ ] 전환율의 분모(세션/유저/노출)와 기간 기준(캘린더/코호트)이 보고서에 명시되어 있다
- [ ] 퍼널 단계의 판정 이벤트가 실제 태깅과 일치하는지 검증했다 (이벤트 누락·중복 발화 확인)
- [ ] 전환 이벤트의 중복이 제거되어 있다 (재전송, 새로고침 — transaction_id 기준 dedup)
- [ ] 미성숙 코호트가 마스킹되어 있다
- [ ] 개선 주장에 표본 크기·유의성·신뢰구간이 붙어 있다
- [ ] 전환율과 함께 전환수·모수를 병기했다 (분모 축소로 만든 전환율 개선 식별)
- [ ] 봇·테스트 주문이 제외되어 있다

## 주의할 점

- **선별 효과로 만든 개선.** 진입 장벽을 높이면(로그인 강제, 필수 입력 추가) 남은 유저의 전환율은 오른다 — 전환수가 줄었는데 전환율 개선으로 보고되는 전형적 착시. 전환율 변화는 항상 모수 변화와 함께 읽는다.
- **평균 회귀.** 최악의 주를 기준으로 잡은 개선 프로젝트는 아무것도 안 해도 좋아 보인다. 개선 측정의 기준선은 충분히 긴 기간의 평균으로.
- **다중 비교.** 세그먼트 20개를 쪼개면 하나는 우연히 유의하다(5% 유의수준에서 기대값 1개). 세그먼트 탐색에서 나온 차이는 발견이 아니라 가설이다.
- **계절성·프로모션 혼입.** 전환율 시계열의 변동을 제품 변경에 귀속하기 전에 프로모션 캘린더를 대조한다.
- **국소 최적화의 함정.** 단계 전환율을 각각 최적화하는 것이 전체 최적이 아닐 수 있다. 장바구니 문턱을 낮추면 담기 전환율은 오르지만 결제 전환율이 떨어지는 식의 단계 간 트레이드오프를 전체 전환율로 판정한다.

## 뽑아야 하는 인사이트

1. **가장 비싼 병목이 어디인가** — 단계별 이탈 × 모수 × 개선 여지를 곱한 우선순위. "결제 단계 10%p 개선"과 "상세→장바구니 2%p 개선"의 기대 매출을 같은 단위로 비교한다.
2. **어느 세그먼트가 평균을 끌어내리는가** — 병목의 이질성. 전사 UX 개편보다 특정 플랫폼 버그 수정이 답인 경우가 많다.
3. **전환 소요 시간의 구조** — 즉시 전환형인가 숙성형인가. 리타게팅·넛지의 타이밍 설계 근거.
4. **마이크로 전환과 최종 전환의 연결 강도** — 어떤 선행 행동이 최종 전환을 예측하는가. 실험 대리 지표(proxy metric) 선정의 근거.
5. **개선의 누적 기록** — 실험 결과의 아카이브. 조직의 전환율 감각(무엇이 통하고 무엇이 안 통하는가)은 이 기록에서만 쌓인다.

## 한계

- 전환은 측정 가능한 마지막 클릭이지, 의사결정의 전체가 아니다. 오프라인 요인·구전·가격 비교 등 로그 밖의 변수가 전환을 결정하는 비중이 크며, 이 부분은 실험으로도 부분적으로만 통제된다.
- 실험 불가능한 영역(전면 가격 변경, 브랜드)의 전환 분석은 준실험(시계열 단절 분석, 지역 대조)으로 대체해야 하고, 인과 확신도는 그만큼 낮아진다.
