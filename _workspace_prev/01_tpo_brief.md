# 01. TPO 브리핑 — 홈화면 콘텐츠 전략 & UX

## 1. 작업 개요

DGPR 기술 블로그(Jekyll wiki) 홈화면(`index.html`)을 리디자인한다. 현재는 wiki 문서의 최근 변경 목록을 `<ul>` 불렛 형식으로 단순 나열한다. 목표는 불렛 리스트를 제거하고 정보 계층이 명확한 모던한 디자인으로 전환하는 것이다.

- **요청 유형:** UI 개선 (콘텐츠 전략 + 정보 구조 정의)
- **범위:** 홈화면 1개 (`index.html` + `home` 레이아웃). archive/recent도 동일 레이아웃을 공유하므로 회귀 검증 필요.
- **제약:** Jekyll Liquid 문법 유지, 다크모드(`html.dark-mode`) 호환, 모바일 우선 반응형(body `max-width: 800px`), vanilla 스택(HTML/SCSS/JS).

## 3. 홈화면의 목적 (우선순위 순)

1. **최신성 전달** — "이 블로그는 살아있고 꾸준히 업데이트된다"를 첫 화면에서 증명한다. (최근 변경 문서 노출)
2. **탐색 진입점 제공** — 방문자가 관심 주제로 빠르게 이동하도록 카테고리/주제 경로를 연다.
3. **콘텐츠 규모 신뢰 형성** — 전체 문서 수, 카테고리 다양성으로 깊이를 암시한다.
4. **재방문 유도** — 깔끔한 정보 계층으로 스캔 비용을 낮춰 다음 방문 의향을 높인다.

> 홈은 "랜딩 페이지"가 아니라 **"지식 베이스의 색인(index) 겸 최근 활동 피드"** 다. 화려함보다 스캔 가능성(scannability)이 핵심.

---

## 4. 방문자 여정 (User Journey)

```
유입 (검색/재방문/직접)
   │
   ▼
[홈 = 최근 변경 피드 + 카테고리 색인]
   │
   ├─▶ A. 최근 글 카드 클릭 → 개별 문서 (핵심 경로, 70%)
   ├─▶ B. 카테고리/주제 클릭 → 카테고리 색인 페이지
   └─▶ C. "전체 문서 보기" → /recent/ 아카이브
```

- **핵심 전환 액션:** 최근 변경 문서 카드 클릭 (경로 A).
- **이탈 방지 지점:** 첫 스크롤 없이(above the fold) 최소 5~7개 글의 제목·날짜·카테고리가 스캔되어야 한다.

## 5. 정보 구조 (Information Architecture)

홈화면을 위→아래 3개 영역으로 구성한다. 현재는 (B)만 존재.

### (A) [선택] 헤더/인트로 — *경량*
- 블로그 한 줄 정체성 또는 생략. 과한 hero 섹션은 지양(스캔 비용↑).
- **우선순위: LOW** — 디자이너 판단에 위임. 추가 시 1줄 텍스트 수준으로 제한.

### (B) [필수] 최근 변경 문서 피드 — *핵심*
현재 `<ul>` 불렛을 **카드/리스트 아이템**으로 전환. 항목당 정보 계층:

| 요소 | 역할 | 시각 비중 | 데이터 소스 |
|------|------|-----------|-------------|
| **문서 제목** | 1차 (강조) | 큼·볼드 | `doc.title` |
| **변경일** | 2차 (서브텍스트) | 작음·muted | `doc.updated \| date: "%b %d, %Y"` |
| **카테고리/태그** | 2차 (맥락) | 작음·배지/라벨 | `doc.parent` 또는 `doc.tag` *(검증 필요 — 아래 6항)* |

- 항목 수: 현행 30개 유지 (초기 노출량은 디자이너가 above-the-fold 기준으로 권고).
- **불렛 마커 제거**, 항목 간 구분은 여백/구분선/카드로 처리.
- 호버/포커스 상태 명확화 (전체 카드 클릭 가능 — 기존 `<a>`가 메타 전체를 감싸는 구조 유지).

### (C) [필수] 전체 문서 진입 링크
- 현행 "View all documents (N items)" 유지. 카운트는 신뢰 신호이므로 존속.
- 위치: 피드 하단. 시각적으로 1차 액션이 아님을 명확히(보조 링크 톤).

### (D) [권고] 카테고리 색인 — *제안*
- 14개 top-level 카테고리(ai-agent, java, jpa, jvm, kafka, msa, system-design, data-engineering, database, design-pattern, code-architecture, springboot, essay, retrospect)로 탐색 경로(여정 B) 강화.
- **우선순위: MEDIUM** — 이번 리디자인 범위에 포함할지 디자이너/오케스트레이터가 결정. 포함 시 칩(chip) 목록 형태 권고.

## 6. 데이터/구현 확인 사항 (검증 필요)

> Frontend 구현 전 반드시 확인. 추측 금지.

1. **카테고리/태그 표시 가능 여부** — wiki 문서 frontmatter는 `parent: [[/java]]`와 `tag: java` 형식. 홈 피드에 카테고리 배지를 넣으려면 `doc.parent`(Liquid에서 `[[/java]]` 문자열 파싱 필요) 또는 `doc.tag` 중 무엇이 일관적으로 채워져 있는지 확인 필요. **태그가 비어있는 문서(`tag: ` 공란) 다수 존재** → 배지가 빈 값으로 깨지지 않도록 fallback 처리 필수.
2. **`summary` 필드** — 일부 frontmatter에 `summary:` 존재하나 대부분 공란. 카드에 요약을 넣을 경우 빈 값 처리 필요. **현 단계 권고: 요약 미노출** (제목+날짜+카테고리로 충분).
3. **category 레이아웃 문서 제외** — 현행 필터(`doc.layout != 'category'`, `doc.title != 'wiki'`) 유지. 카테고리 인덱스 문서가 피드에 섞이지 않도록 보존.
4. **archive/recent 회귀** — 동일 `home` 레이아웃·`.post-list` 클래스 사용. SCSS 변경 시 두 페이지 함께 확인.

## 7. SEO / 메타 고려사항

- 홈은 카테고리 키워드 허브. 카테고리 배지/링크는 내부 링크 밀도를 높여 색인에 유리(여정 D 채택 시 부가 효과).
- 제목 텍스트는 실제 `<a>` 앵커 텍스트로 유지(이미지·아이콘만으로 대체 금지) — 접근성·SEO 동시 충족.
- 핵심 키워드군(기존 태그 기반): `java`, `jvm`, `jpa`, `kafka`, `msa`, `system-design`, `data-engineering`, `rag`, `llm`, `agent`, `redis`, `os`, `algorithm`, `security`, `design-pattern`.

---

## 8. 품질 기준 (Designer/Frontend 인계용)

- [ ] 불렛 마커 완전 제거, 정보 계층 = 제목(강조) > 날짜/카테고리(서브)
- [ ] 다크모드에서 대비·가독성 유지 (`html.dark-mode` 검증)
- [ ] 모바일 375px에서 카드/항목 레이아웃 깨짐 없음
- [ ] 전체 카드 클릭 영역 + 명확한 hover/focus 상태 (키보드 접근성 포함, WCAG AA)
- [ ] Liquid 필터·조건문 보존 (category 문서 제외, public 필터)
- [ ] archive/recent 페이지 회귀 없음
- [ ] 빈 메타값(태그/요약 공란) fallback 처리
