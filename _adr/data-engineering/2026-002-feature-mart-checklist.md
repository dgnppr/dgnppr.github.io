## Context

피쳐 마트는 데이터 마트의 특수형이다. 일반 마트 체크리스트([[2026-001-data-mart-checklist]])의 항목 — 그레인, 멱등성, 백필, 품질 테스트 — 은 전부 그대로 적용된다. 그러나 그것만으로는 부족하다.

BI 마트가 틀리면 숫자가 틀렸다는 문의가 온다. 피쳐 마트가 틀리면 문의가 오지 않는 경우가 많다. 모델이 조용히 나빠지거나, 더 다루기 어려운 형태로는 오프라인 평가에서만 좋아 보인다. 피쳐 마트 특유의 사고는 세 가지로 수렴한다. **미래 데이터 누수(leakage)**, **point-in-time 위반**, **training-serving skew**다. 셋 다 만드는 시점에는 드러나지 않고, 모델이 프로덕션에 나간 뒤에 나타난다.

여기에 하나를 더 얹는다. **학습셋 재현성**이다. 어제 뽑은 학습 데이터와 오늘 다시 뽑은 학습 데이터가 다르면, 실험 A와 실험 B의 성능 차이가 피쳐 때문인지 데이터가 바뀐 탓인지 판정할 수 없다. 누수·skew가 "틀린 모델"을 만든다면, 재현성 부재는 "무엇이 옳은지 판단할 수 없는 상태"를 만든다.

이 문서는 일반 마트 체크리스트 위에 얹는 피쳐 마트 전용 게이트다. 항목을 기계적으로 다 채우는 것이 목적이 아니라, 소비 형태(1단계)에 따라 적용 범위를 정하는 것이 목적이다.

## Decision

신규 피쳐 마트는 [[2026-001-data-mart-checklist]]의 8단계를 먼저 통과한 뒤, 아래 피쳐 특화 체크리스트를 추가로 통과해야 배포한다.

---

## 1. 소비자·시점 정의

- [ ] **소비 모델과 사용 형태를 특정했다.** 학습용(offline)인지, 서빙용(online)인지, 둘 다인지. 배치 스코어링인지 실시간 추론인지. 이 답이 아키텍처 전체를 결정한다. 오프라인 전용이면 테이블 하나로 끝나지만, 온라인 서빙이 있으면 동기화 문제(7단계)가 통째로 추가된다.
- [ ] **예측 시점(prediction time)을 정의했다.** "어느 시점에 서 있는 관찰자가 아는 값인가"가 피쳐 마트의 근본 질문이다. 예측 시점이 정의되지 않은 피쳐 마트는 누수 여부를 판정할 기준 자체가 없다.
- [ ] **스코어링 주기와 피쳐 갱신 주기의 정합을 확인했다.** 매시간 스코어링하는 모델에 일 1회 갱신 피쳐를 물리면, 하루 중 23시간은 낡은 피쳐로 추론한다. 그것이 허용되는지는 모델 성능의 문제이므로 ML 엔지니어와 명시적으로 합의한다.
- [ ] **라벨 윈도우를 피쳐 윈도우와 분리해 정의했다.** 예: 피쳐는 예측 시점 이전 30일, 라벨은 예측 시점 이후 7일. 두 윈도우가 겹치면 그 자체가 누수다.
- [ ] **라벨 자체의 품질을 검증했다.** 라벨 확정에 걸리는 지연(구매 확정, 환불 마감), 라벨이 뒤바뀌는 케이스(취소 후 재구매), 관측 기간 미달로 라벨을 판정할 수 없는 엔티티(censoring)의 처리 규칙. 라벨이 흔들리면 피쳐가 완벽해도 모델은 노이즈를 학습한다. 클래스 불균형 비율도 이 시점에 기록해둔다.
- [ ] **기존 피쳐를 먼저 뒤졌다.** 같은 엔티티의 유사 피쳐가 이미 있으면 재사용한다. 피쳐 중복은 마트 중복보다 다루기 어렵다. 비슷하지만 미묘하게 다른 피쳐 두 개는 모델 디버깅의 원인 후보를 늘린다.

## 2. 엔티티·그레인

- [ ] **엔티티와 그레인을 한 문장으로 썼다.** "한 행은 하나의 (user_id, feature_date) 조합이며, 해당 날짜 00:00 KST 시점에 관찰 가능한 값이다." 피쳐 마트의 그레인 문장에는 반드시 **관찰 시점**이 포함돼야 한다.
- [ ] **spine을 먼저 설계했다.** 학습셋 생성의 뼈대는 (entity_id, prediction_ts, label)의 spine 테이블이다. spine을 먼저 확정하고 피쳐를 spine에 as-of 조인으로 붙이는 구조로 만들면, 누수 검증이 "spine 조인이 올바른가" 하나로 수렴한다. 피쳐 테이블마다 제각각 라벨을 붙이는 구조는 누수 검증 지점이 피쳐 수만큼 늘어난다. spine의 최소 형태는 다음과 같다.

```sql
-- 학습셋 spine: 한 행 = 하나의 예측 이벤트
CREATE TABLE feat.spine_churn_v1 (
  entity_id      STRING    NOT NULL,   -- 예측 대상 (user_id)
  prediction_ts  TIMESTAMP NOT NULL,   -- 예측이 서는 시점 (as-of 컷 기준)
  label          INT64,                -- 예측 시점 이후 라벨 윈도우에서 확정
  label_ts       TIMESTAMP,            -- 라벨이 확정된 시각 (재현성·검증용)
  split          STRING                -- train / valid / test (시간 기준)
)
PARTITION BY DATE(prediction_ts);
-- 유니크 키: (entity_id, prediction_ts) — 유니크 테스트로 강제한다.
```

- [ ] **엔티티 ID 체계가 학습과 서빙에서 동일한지 확인했다.** 학습 데이터는 회원 ID 기준인데 서빙 요청은 디바이스 ID로 들어온다면, 그 피쳐는 서빙 시점에 조회 자체가 안 된다. ID 매핑(비회원 → 회원 전환 등)의 시점 처리도 포함한다.
- [ ] **엔티티 생명주기를 정의했다.** 탈퇴·휴면·차단 엔티티를 피쳐 마트에 계속 실을지, 서빙 요청이 올 수 없는 엔티티를 계산에서 제외할지. 전체 엔티티를 매일 계산하면 비용의 대부분이 추론될 일 없는 엔티티에 쓰이는 경우가 흔하다.

## 3. Point-in-time correctness

가장 중요한 단계다. 여기서 실패하면 나머지가 다 맞아도 모델은 프로덕션에서 배신한다.

- [ ] **모든 피쳐가 "그 시점에 알 수 있었던 값"인지 검증했다.** 조건은 하나다. `feature_ts <= prediction_ts`. 집계 피쳐라면 집계 윈도우의 상한이 예측 시점을 넘지 않아야 한다.
- [ ] **as-of 조인으로 피쳐를 붙였다.** 각 spine 행의 `prediction_ts` 이하에서 가장 최근 피쳐 값을 가져오는 것이 as-of 조인의 정의다. BigQuery에서는 조건 조인 후 윈도우 함수로 최신 1건을 고르는 패턴이 명시적이고 검증하기 쉽다.

```sql
-- as-of 조인: prediction_ts 이하 최신 스냅샷 1건
WITH joined AS (
  SELECT
    s.entity_id,
    s.prediction_ts,
    s.label,
    f.tier,
    f.feature_ts,
    ROW_NUMBER() OVER (
      PARTITION BY s.entity_id, s.prediction_ts
      ORDER BY f.feature_ts DESC        -- 예측 시점에 가장 가까운 과거값
    ) AS rn
  FROM feat.spine_churn_v1 AS s
  LEFT JOIN dim.user_tier_snapshot AS f
    ON f.user_id = s.entity_id
   AND f.feature_ts <= s.prediction_ts  -- 미래 스냅샷 차단 (핵심 조건)
)
SELECT entity_id, prediction_ts, label, tier
FROM joined
WHERE rn = 1;
-- 함정: ON 절의 부등식을 빼고 = 로만 조인하면 최신값 테이블과 붙어 미래가 샌다.
-- 함정: 스냅샷 소스가 크면 이 조인은 풀스캔 셔플이 된다. prediction 날짜 기준으로
--        스냅샷을 파티션 프루닝하도록 날짜 범위를 먼저 좁힌다.
```

- [ ] **"알 수 있었던 시점"을 발생 시각이 아니라 도착 시각으로 판정했다.** 이벤트가 월요일에 발생해도 파이프라인에 수요일 도착했다면, 화요일 예측 시점의 모델은 그 값을 알 수 없었다. 지각이 심한 소스는 `event_ts`가 아니라 적재 시각(availability time) 기준으로 컷해야 학습과 서빙의 조건이 같아진다. 이 구분을 무시하면 오프라인에서만 존재하는 미래 정보가 생긴다.
- [ ] **타임존 경계를 일치시켰다.** `prediction_ts`, 피쳐 윈도우 경계, 라벨 윈도우 경계가 서로 다른 타임존이면 하루 단위 어긋남이 조용히 생긴다. 저장은 UTC로 통일하고, "일별" 경계가 KST 자정을 의미한다면 윈도우 계산에서 `TIMESTAMP_SUB(prediction_ts, INTERVAL 30 DAY)`가 UTC 기준인지 KST 기준인지 명시한다. 서머타임이 있는 지역이 섞이면 "30일"이 실제로 30일이 아닐 수 있으므로, 경계는 날짜가 아니라 타임스탬프로 계산하는 것이 안전하다.
- [ ] **차원 속성 조인이 as-of 조인인지 확인했다.** 유저 등급, 상품 카테고리 같은 속성을 **현재값 테이블**과 조인하면, 과거 예측 시점의 학습 행에 미래의 속성이 붙는다. SCD Type 2 또는 일별 스냅샷과 예측 시점 기준으로 조인해야 한다(위 as-of 패턴).
- [ ] **mutable 소스를 식별하고 스냅샷을 확보했다.** in-place update되는 컬럼(주문 상태, 잔액, 등급)은 소스에 과거값이 없다. 스냅샷 축적 없이는 point-in-time 재구성이 불가능하다. 이 피쳐가 필요하면 스냅샷부터 시작해야 하고, 스냅샷 시작 이전 기간은 학습 데이터로 쓸 수 없다.
- [ ] **스냅샷 주기가 피쳐의 시간 해상도와 맞는지 확인했다.** 일별 스냅샷 기반 피쳐는 하루 안의 변화를 표현할 수 없다. 시간 단위 예측이 필요한 모델에 일별 스냅샷 피쳐를 물리면, as-of 조인이 맞아도 사실상 낡은 값이다.
- [ ] **지각 데이터·백필이 학습 재현성을 깨는지 점검했다.** 어제 만든 학습 데이터와 오늘 다시 뽑은 학습 데이터가 다르면 실험 비교가 불가능하다. 피쳐 마트의 과거 파티션은 immutable로 취급하는 것이 기본값이고, 재적재가 필요하면 버전을 남긴다.
- [ ] **학습셋 생성의 재현 정보를 실험 트래킹에 기록했다.** 최소한 (a) 학습 데이터 생성 쿼리 또는 그 커밋 해시, (b) 참조한 피쳐 마트의 버전·스냅샷 날짜, (c) spine의 필터 조건과 기간을 실험 메타데이터로 남긴다. 이것이 없으면 "이 모델은 정확히 어떤 데이터로 학습됐는가"에 답할 수 없고, 좋은 실험을 재현하지도 나쁜 실험을 규명하지도 못한다.

## 4. 누수(leakage) 점검

- [ ] **라벨의 프록시 피쳐가 없는지 소거했다.** 취소 예측 모델에 "환불 처리 여부", 이탈 예측에 "마지막 접속 후 경과일을 라벨 윈도우 포함으로 계산한 값" 같은 것. 피쳐 목록을 놓고 "이 값이 라벨 확정 과정에서 생성되는가"를 하나씩 묻는다.
- [ ] **집계 윈도우가 라벨 기간을 침범하지 않는지 확인했다.** "최근 30일 구매 횟수"의 30일이 예측 시점 기준인지, 적재 시점 기준인지. 적재 시점 기준 rolling 집계를 그대로 학습에 쓰면 누수다.
- [ ] **분할(split) 방식이 시간 순서를 존중하는지 확인했다.** 시계열 성격의 문제에 랜덤 분할을 쓰면 미래 행이 train에, 과거 행이 test에 들어간다. 피쳐가 완벽해도 분할 자체가 누수다. 시간 기준 분할(train은 과거, test는 미래)이 기본값이다.
- [ ] **그룹 누수를 확인했다.** 같은 엔티티(유저)의 행이 train과 test에 나뉘어 들어가면, 모델은 일반화가 아니라 그 유저를 암기한 성능을 보여준다. 문제 특성에 따라 엔티티 단위 분할을 검토한다.
- [ ] **전처리 누수를 확인했다.** 정규화 통계(평균·표준편차), 인코딩 사전, 결측 대치값을 **전체 기간·전체 데이터로 fit**하면 test 정보가 train에 스며든다. fit은 train에서만, transform은 양쪽에 적용한다. 이 규칙은 피쳐 마트가 아니라 학습 파이프라인의 책임이지만, 마트가 "전체 기간 통계 컬럼"을 제공하는 순간 마트의 책임이 된다.
- [ ] **누수 자동 테스트를 걸었다.** 코드 리뷰가 아니라 테스트로 막는다. 최소 두 개를 상시 실행한다. 하나는 시점 위반, 하나는 윈도우-라벨 겹침이다.

```sql
-- 테스트 1: 미래 피쳐 유입 (0건이어야 통과)
SELECT COUNT(*) AS violations
FROM feat.training_set_churn_v1
WHERE feature_ts > prediction_ts;

-- 테스트 2: 집계 윈도우 상한이 예측 시점을 넘음 (0건이어야 통과)
SELECT COUNT(*) AS violations
FROM feat.training_set_churn_v1
WHERE window_end_ts > prediction_ts;
```

- [ ] **오프라인 지표가 비정상적으로 좋으면 축하 전에 조사했다.** AUC가 갑자기 뛰면 피쳐가 좋아진 것보다 누수가 생겼을 가능성이 높다. 판단을 감이 아니라 실험으로 남긴다. 신규 피쳐 추가 시 그 피쳐를 뺀 대조 실험(on/off)을 함께 돌려 기여도를 정량화하고, 피쳐 중요도 상위에 갑자기 올라온 피쳐는 생성 경로를 역추적한다. 단일 피쳐가 중요도를 지배하면 라벨 프록시일 확률이 높다.

## 5. 피쳐 정의·명세

- [ ] **피쳐 명세를 코드와 함께 남겼다.** 최소 필드: 이름, 수식(또는 소스 코드 링크), 집계 윈도우, 단위, 갱신 주기, NULL의 의미, 오너, 소비 모델 목록, 생성일. 명세 없는 피쳐는 만든 사람이 떠나면 재현하기 어려운 블랙박스가 된다. 명세는 별도 문서가 아니라 코드(yaml, 데코레이터, dbt meta)에 붙여서 배포와 함께 굴러가게 한다.
- [ ] **다중 윈도우 피쳐를 매크로로 일관되게 생성했다.** 같은 지표의 7d/30d/90d 버전을 손으로 세 번 쓰면 언젠가 한 창의 경계 조건이 어긋난다. 윈도우 목록을 파라미터로 받는 매크로(dbt 매크로, Jinja 반복, 코드 생성)로 뽑아 경계 로직을 한 곳에 둔다. 그래야 "30일은 inclusive인데 7일은 exclusive"같은 미묘한 불일치가 구조적으로 막힌다.

```sql
-- 다중 윈도우를 한 정의에서: 윈도우 경계 로직이 한 곳에만 존재한다
SELECT
  entity_id,
  prediction_ts,
  COUNTIF(event_ts >  TIMESTAMP_SUB(prediction_ts, INTERVAL 7  DAY)) AS user__order_cnt__7d,
  COUNTIF(event_ts >  TIMESTAMP_SUB(prediction_ts, INTERVAL 30 DAY)) AS user__order_cnt__30d,
  COUNTIF(event_ts >  TIMESTAMP_SUB(prediction_ts, INTERVAL 90 DAY)) AS user__order_cnt__90d
FROM feat.spine_churn_v1 AS s
JOIN raw.order_events AS e USING (entity_id)
WHERE e.event_ts <= s.prediction_ts     -- 공통 상한: 예측 시점
  AND e.event_ts >  TIMESTAMP_SUB(s.prediction_ts, INTERVAL 90 DAY)  -- 프루닝
GROUP BY entity_id, prediction_ts;
```

- [ ] **NULL과 0과 미관측을 구분했다.** "구매 이력이 없어서 0"과 "데이터가 유실돼서 NULL"과 "신규 유저라 관측 기간 미달"은 다른 값이다. 뭉개서 0으로 채우면 모델이 데이터 결함을 패턴으로 학습한다. 기본값 대치는 마트가 아니라 학습 파이프라인에서, 명시적으로 한다.
- [ ] **네이밍에 엔티티·의미·윈도우를 넣었다.** 예: `user__order_cnt__30d`, `item__view_to_order_rate__7d`. 이름만 보고 그레인과 윈도우를 추론할 수 있어야 한다.
- [ ] **단위와 스케일을 통일했다.** 금액은 통화 명시, 시간은 초/분/일 중 하나로, 비율은 0~1인지 퍼센트인지. 같은 마트 안에서 `_sec`과 `_min`이 섞이면 언젠가 단위 환산 사고가 난다.
- [ ] **피쳐 로직 변경은 교체가 아니라 버전 추가로 했다.** 기존 피쳐의 수식을 조용히 바꾸면 과거에 학습된 모델의 서빙 입력이 달라진다. `_v2`로 새로 만들고, 구버전은 의존 모델이 사라진 뒤 폐기한다.

## 6. 적재·백필·계산 비용

- [ ] **과거 백필이 point-in-time으로 가능한지 판정했다.** 소스가 append-only 이벤트라면 과거 예측 시점 기준 재계산이 가능하다. mutable 스냅샷 기반이라면 스냅샷이 존재하는 기간만 백필 가능하다.
- [ ] **재구성 불가능한 과거는 백필하지 않기로 했다.** 현재값으로 과거를 근사해 백필하면 그 기간의 학습 데이터는 전부 오염이다. 그냥 축적 시작 시점부터 쓰는 것이 낫다. 데이터가 적은 것은 한계지만, 왜곡된 것은 부채다.
- [ ] **백필한 구간에 플래그를 남겼다.** 실시간 축적분과 백필 재구성분은 미묘하게 다를 수 있다(지각 데이터 반영 범위, 소스 상태). `_is_backfilled` 같은 플래그를 남겨두면 나중에 모델 성능이 특정 기간에서만 이상할 때 원인 후보를 하나 줄일 수 있다.
- [ ] **윈도우 집계의 재계산 구조를 비용 기준으로 선택했다.** 롤링 윈도우 피쳐를 매일 전체 기간 재계산하면 구현은 단순하지만 스캔 비용이 매일 윈도우 길이에 비례해 반복된다. 90일 윈도우를 매일 전체 재계산하면 같은 90일을 매일 다시 스캔하는 셈이다. 대안은 증분 누적 구조(전일 상태 + 당일 델타)지만, 윈도우에서 빠지는 만료분 처리와 지각 데이터 반영이 복잡해지고 재현성 검증이 어려워진다. 판단 기준은 명확하다. 데이터가 작거나 윈도우가 짧으면 전체 재계산이 정답이고, 데이터가 크고 윈도우가 길며 매일 도는 피쳐만 증분을 도입한다. 증분은 명백히 비용이 문제일 때 지불하는 복잡도다.

| 방식 | 장점 | 비용 | 언제 |
|------|------|------|------|
| 매일 전체 재계산 | 단순·멱등·재현 쉬움 | 매일 윈도우 전체 스캔 반복 | 데이터 작음, 윈도우 짧음, 대부분의 경우 |
| 증분 누적 | 스캔 비용 절감 | 만료분·지각 데이터 처리 복잡, 재현성 검증 어려움 | 데이터 큼 + 긴 윈도우 + 매일 실행 |

- [ ] **스냅샷 보존 기간과 비용을 계산했다.** 일별 전체 스냅샷은 스토리지가 선형으로 늘어난다. 보존 기간, 파티션 만료, 저압축 컬럼 제외를 설계에 포함한다.

## 7. 온라인·오프라인 일관성 (온라인 서빙이 있는 경우)

- [ ] **변환 로직이 한 벌인지 확인했다.** 오프라인은 SQL, 온라인은 애플리케이션 코드로 같은 피쳐를 각각 구현하면 skew는 필연이다. 오프라인에서 계산해 온라인 스토어로 동기화하거나, 동일 변환 정의를 양쪽이 공유하는 구조로 만든다.
- [ ] **온라인 피쳐의 신선도 SLA를 정의했다.** 배치 동기화 주기가 하루면, 서빙 시점의 피쳐는 최대 하루 낡은 값이다. 그 staleness가 모델 성능에 허용 가능한지 소비자(ML 엔지니어)와 합의한다. TTL을 걸었다면 만료 시 동작(폴백 vs 에러)도 함께 정한다.
- [ ] **조회 실패·미존재 엔티티의 폴백을 정의했다.** 신규 유저(피쳐 없음), 스토어 장애(조회 실패), 부분 실패(일부 피쳐만 조회됨) 각각에 대해 모델이 받을 값을 정한다. 폴백 기본값은 학습 데이터의 결측 처리와 **같은 값**이어야 한다. 학습은 NULL로, 서빙은 0으로 폴백하면 그 자체가 skew다.
- [ ] **온라인 스토어 동기화의 성공·완결성을 감지했다.** 배치 → 온라인 스토어 동기화는 조용히 실패하거나 부분만 반영되는 경우가 흔하다. 동기화 후 (a) 반영된 엔티티 건수가 배치 산출 건수와 일치하는지, (b) 최신 반영 타임스탬프가 SLA 안인지, (c) 오프라인 값과 온라인 값을 표본 대조했을 때 일치하는지를 자동 검사한다. 부분 동기화는 "일부 유저만 낡은 피쳐로 추론"이라는 형태로 나타나고, 감지 장치가 없으면 성능 저하로만 뒤늦게 드러난다.
- [ ] **서빙 레이턴시 예산을 확인했다.** 온라인 조회의 p99 레이턴시가 추론 SLA 안에 들어가는지. 피쳐 수가 늘수록 조회 비용이 늘어나므로, 모델별 피쳐 조회를 배치화(한 번의 lookup)하는 구조를 확인한다.
- [ ] **skew 측정을 자동화했다.** 서빙 시점에 실제로 모델에 들어간 피쳐 값을 로깅하고, 같은 (엔티티, 시점)의 오프라인 재계산 값과 주기적으로 대조한다. skew는 "없다고 믿는" 것이 아니라 측정하는 것이다. 신규 모델은 섀도우 모드(예측만 하고 반영 안 함)로 일정 기간 돌려 서빙 입력 분포를 학습 분포와 먼저 비교한다.

## 8. 모니터링·거버넌스

- [ ] **분포 모니터링을 걸었다.** 기본 품질 테스트(유니크·not null·신선도·row count) 위에 피쳐별 mean, null rate, cardinality의 drift 감지를 추가한다. 분포 비교 지표(PSI 등)를 쓰면 임계치 관리가 일관돼진다. 피쳐 마트의 장애는 값이 없어지는 형태보다 분포가 조용히 틀어지는 형태로 오는 경우가 많다.
- [ ] **물리적으로 불가능한 값의 가드를 걸었다.** 음수 나이, 1을 넘는 비율, 미래 날짜처럼 도메인상 불가능한 값은 분포 감지보다 싸고 확실한 하드 가드로 잡는다.
- [ ] **피쳐 커버리지를 추적했다.** 전체 엔티티 대비 non-null 비율. 커버리지가 조용히 떨어지면 모델은 기본값으로 추론하는 비율이 늘어난 것이고, 이것도 성능 저하로 나타난다.
- [ ] **피쳐 중요도의 급변을 모니터링했다.** 재학습마다 상위 피쳐 중요도를 기록하고, 특정 피쳐의 기여도가 갑자기 급등·급락하면 알린다. 급등은 신규 누수 또는 소스 변경, 급락은 소스 파이프라인 고장이나 커버리지 하락의 신호인 경우가 많다. 중요도 변화는 모델 성능이 무너지기 전에 오는 선행 지표로 쓸 수 있다.
- [ ] **드리프트 감지와 재학습 트리거를 연결했다.** 분포가 임계치를 넘으면 누구에게 알리고, 재학습을 자동으로 걸지 사람이 판단할지. 감지만 하고 액션이 없는 모니터링은 대시보드 장식이다.
- [ ] **피쳐-모델 의존성을 기록했다.** 어떤 모델이 어떤 피쳐를 쓰는지 등록 없이는 피쳐 폐기·변경 판단이 불가능하다. 최소한 소비 모델 목록을 명세에 남긴다. 어떤 모델도 안 쓰는 피쳐는 계산 비용만 내는 부채이므로 주기적으로 정리한다.
- [ ] **모델에 사용 금지인 속성을 소거했다.** 규제·정책상 모델 입력으로 쓸 수 없는 속성(민감정보, 차별 우려 변수)이 파생 피쳐로 스며들지 않았는지. 원본 컬럼만이 아니라 그것으로 만든 파생값까지 본다.

## 경험칙 요약

1. **오프라인 지표가 너무 좋으면 누수부터 의심한다.** 피쳐 마트 운영에서 가장 값비싼 교훈은 "성능이 좋아 보였던" 모델에서 나오는 경우가 많다.
2. **spine 먼저, 피쳐는 나중이다.** (entity_id, prediction_ts, label) 뼈대를 확정하고 피쳐를 as-of로 붙이면 누수 검증 지점이 하나로 수렴한다.
3. **mutable 컬럼은 스냅샷 없이 피쳐로 쓰지 않는다.** 현재값 테이블과의 조인은 미래를 보고 과거를 학습시키는 가장 흔한 경로다.
4. **랜덤 분할은 시계열 데이터의 조용한 누수다.** 시간 기준 분할이 기본값이고, 랜덤 분할은 정당화가 필요한 예외다.
5. **재구성한 과거는 대부분 왜곡이다.** point-in-time 백필이 불가능하면 백필하지 말고 오늘부터 쌓는다.
6. **피쳐는 만든 사람보다 오래 산다.** 수식·윈도우·NULL 의미가 명세로 남지 않은 피쳐는 언젠가 아무도 못 건드리는 블랙박스가 된다.
7. **변환 로직 두 벌이면 skew는 예정된 사고다.** 온라인·오프라인을 따로 구현했다면 서빙 입력 로깅과 대조 측정 자동화가 최소한의 방어다.
8. **폴백 기본값은 학습의 결측 처리와 같아야 한다.** 학습은 NULL, 서빙은 0이면 그 자체가 skew다.
9. **NULL을 0으로 뭉개면 모델이 데이터 결함을 학습한다.** 미관측·결측·진짜 0의 구분은 피쳐 설계의 일부다.
10. **피쳐 로직은 고치는 게 아니라 버전을 올리는 것이다.** 조용한 수식 변경은 이미 배포된 모델에 대한 무통보 입력 변경이다.
11. **재현할 수 없는 학습셋은 실험 결과를 판단 불가능하게 만든다.** 생성 쿼리·마트 버전·스냅샷 날짜를 실험 트래킹에 남기는 것이 재현성의 최소 조건이다.
12. **다중 윈도우는 경계 로직을 한 곳에 모은다.** 7d/30d/90d를 손으로 각각 쓰면 경계 불일치는 시간 문제다.
13. **온라인 동기화는 실패보다 부분 반영이 더 다루기 어렵다.** 건수·타임스탬프·표본 대조를 자동 검사하지 않으면 부분 동기화는 성능 저하로만 드러난다.

## Consequences

- 일반 마트 대비 구축 비용이 명확히 늘어난다. 특히 point-in-time 검증과 스냅샷 축적은 선불이 크다. 대신 "오프라인에서만 좋은 모델"이라는 가장 비싼 실패를 구조적으로 차단한다.
- mutable 소스 기반 피쳐는 스냅샷 시작 전 기간을 학습에 못 쓰게 된다. 데이터 양의 손해를 감수하는 결정이며, 이는 의도된 트레이드오프다.
- 재현성 기록(생성 쿼리·마트 버전·스냅샷 날짜)과 실험 트래킹 연결은 초기에 추가 작업이지만, 실험 수가 늘수록 이득이 커진다. 기록이 없으면 좋은 실험을 재현하지도 나쁜 실험을 규명하지도 못한다.
- 증분 누적 구조는 계산 비용을 줄이지만 재현성·정확성 검증을 어렵게 한다. 비용이 명백한 문제일 때만 도입하고, 기본값은 전체 재계산이다.
- 온라인 서빙이 없는 조직에서는 7단계가 통째로 불필요하다. 체크리스트를 기계적으로 다 채우는 것이 아니라 소비 형태(1단계)에 따라 적용 범위를 정한다.
- 분할·전처리 누수(4단계 일부)는 엄밀히는 학습 파이프라인의 책임 영역이다. 그래도 이 문서에 두는 이유는, 피쳐 마트 설계자가 이 함정을 모르면 "전체 기간 통계 컬럼" 같은 누수 유발 피쳐를 마트에 넣기 때문이다.

## Alternatives Considered

1. **일반 데이터 마트 체크리스트만 재사용**: 그레인·멱등성·품질은 커버하지만 누수·point-in-time·skew·재현성이라는 피쳐 마트의 핵심 사고를 막지 못한다 — 미채택.
2. **피쳐 스토어 도구 도입으로 대체(Feast, Vertex AI Feature Store 등)**: 도구는 point-in-time join과 온라인 동기화를 해결해주지만, 라벨 프록시 누수·피쳐 명세·버저닝 규율·거버넌스·학습셋 재현성은 도구가 해결하지 못한다. 도구 도입 여부와 무관하게 체크리스트가 선행돼야 한다 — 부분 채택(도구 도입은 별도 ADR로 판단).
3. **모델별 학습 파이프라인 안에서 피쳐를 그때그때 계산**: 재사용이 안 되고 같은 피쳐가 파이프라인마다 미묘하게 다르게 구현된다. 소규모 초기에는 유효하지만 모델이 3개를 넘으면 관리가 어려워진다 — 미채택.
