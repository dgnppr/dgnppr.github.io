## Context

데이터 마트를 만드는 일은 반복 업무다. 그러나 만들 때마다 품질 편차가 크다. 그레인이 문서화되지 않은 마트, 백필이 불가능한 마트, 지표 수식이 BI 도구와 SQL 두 곳에 존재하는 마트가 계속 생긴다. 사고는 대부분 마트를 만드는 시점이 아니라 운영 3~6개월 후에 터지고, 그 시점에는 만든 사람의 컨텍스트가 이미 사라져 있다.

마트 구축의 판단 기준을 개인 기억이 아니라 체크리스트로 고정한다. 이 문서는 신규 마트를 만들 때 통과해야 하는 게이트이자, 운영 경험에서 나온 노하우 모음이다.

## Decision

신규 데이터 마트는 아래 8단계 체크리스트를 통과한 뒤 배포한다. 순서가 중요하다 — 특히 1~2단계(요구사항·그레인)를 건너뛰고 SQL부터 쓰는 것을 금지한다.

---

## 1. 요구사항 정의

- [ ] **소비자를 특정했다.** 대시보드, 분석가의 애드혹 쿼리, ML 피처, 역방향 ETL 중 무엇인지. 소비자가 불명확한 마트는 만들지 않는다.
- [ ] **일회성인지 반복 소비인지 구분했다.** 한 번 쓰고 버릴 질문이면 마트가 아니라 애드혹 쿼리로 답한다. 마트는 반복 소비가 확인된 질문에만 만든다 — 일회성 요청마다 마트를 만든 조직은 1년 뒤 아무도 안 쓰는 테이블 수십 개를 유지보수하게 된다.
- [ ] **요청자에게 컬럼 목록이 아니라 최종 쿼리(또는 결과 표본)를 받았다.** "이런 컬럼이 필요해요"는 요구사항이 아니다. 요청자가 마트 위에서 실제로 돌릴 쿼리, 또는 최종적으로 보고 싶은 표의 모양을 받아야 그레인과 집계 수준을 역산할 수 있다. 이때 "이 숫자로 어떤 의사결정을 하는가"까지 물으면 요구사항의 절반은 저절로 정리된다.
- [ ] **지표 정의를 서면으로 합의하고 지표마다 오너를 지정했다.** "매출"이라는 단어 하나에도 결정할 것이 많다 — 환불 포함 여부, VAT 포함 여부, 취소 건의 귀속 시점(주문일 vs 취소일), 통화 환산 기준. 정의가 흔들릴 때 최종 판정할 사람(지표 오너)이 없으면 정의는 부서마다 다르게 진화한다.
- [ ] **기존 마트로 해결 가능한지 먼저 확인했다.** 비슷한 마트가 이미 있다면 컬럼 추가가 신규 구축보다 낫다. 경험상 신규 구축 요청의 절반은 기존 마트 확장으로 해결된다.
- [ ] **신선도 SLA와 소비 주기를 합의했다.** "매일 오전 9시까지 전일 데이터"처럼 시각으로 명시한다. 대시보드 리프레시 주기·회의 시각과 역산해서 잡아야 의미가 있다 — 9시 회의에 쓰는 데이터의 SLA가 9시면 이미 늦은 것이다. SLA 미충족 시 누구에게 어떤 채널로 알릴지도 이 시점에 정한다.

## 2. 그레인 정의

가장 중요한 단계다. 마트 사고의 대부분은 그레인 모호성에서 나온다.

- [ ] **"이 테이블의 한 행은 ___이다"를 한 문장으로 썼다.** 예: "한 행은 하나의 (주문일, 상품 ID) 조합이다." 이 문장을 테이블 description(스키마 메타데이터)에 그대로 박는다 — 위키가 아니라 테이블에. 위키는 낡지만 description은 조회하는 사람 눈앞에 있다.
- [ ] **팩트 유형을 의식적으로 선택했다.** 트랜잭션 팩트(발생 건별), 주기 스냅샷(일별 잔액·재고), 누적 스냅샷(주문 → 결제 → 배송 마일스톤) 중 무엇인지. 요구가 "상태의 추이"면 스냅샷, "발생의 집계"면 트랜잭션이다. 누적 스냅샷은 행이 업데이트되는 테이블이라 적재 전략(5단계)이 완전히 달라진다.
- [ ] **그레인이 두 개면 테이블을 두 개로 나눴다.** 주문 단위 행과 일별 집계 행을 한 테이블에 섞으면 소비자는 언젠가 둘을 합산한다.
- [ ] **유니크 키를 확정하고 테스트로 강제했다.** 그레인 문장에서 도출된 키 조합의 uniqueness 테스트가 없는 마트는 미완성이다.
- [ ] **조인 fan-out을 검증했다.** 1:N 관계 조인으로 행이 불어나 지표가 부풀지 않는지, 조인 전후 row count와 금액 합계를 비교했다. 다대다 관계가 필요하면 조인을 우기지 말고 bridge 테이블로 분리한다.
- [ ] **차원 조인 실패 행을 확인했다.** 팩트에는 있는데 차원에 없는 키(late-arriving dimension, 삭제된 마스터)가 몇 %인지 실측하고, 처리 방침(unknown 버킷 vs 제외 vs 적재 보류)을 정했다.

## 3. 소스 검증

- [ ] **소스의 오너와 변경 통보 채널을 확인했다.** 업스트림 스키마가 바뀔 때 통보받을 방법이 없다면 그 마트는 시한폭탄이다. 통보 채널이 없으면 최소한 스키마 drift 감지(컬럼 목록·타입 스냅샷 비교)를 자동화해서 통보를 만들어낸다.
- [ ] **소스를 프로파일링했다.** 믿지 말고 실측한다: 키 유일성, 컬럼별 NULL률, 중복률, 값 분포(카테고리 값 목록), 일별 row count 추이. 프로파일링 없이 짠 변환 로직은 소스의 "문서에 없는 진실"(빈 문자열로 저장되는 NULL, 테스트 계정, 음수 금액)에 반드시 걸린다.
- [ ] **소스의 업데이트 시맨틱을 파악했다.** append-only인지, in-place update가 있는지, 삭제가 있는지(hard/soft delete), CDC인지 스냅샷인지. 이걸 모르면 적재 전략을 정할 수 없다. 특히 hard delete가 있는 소스를 증분 적재하면 마트에는 지워진 행이 유령으로 남는다.
- [ ] **지각 데이터(late-arriving data)의 분포를 측정했다.** 이벤트가 발생 후 며칠까지 도착하는지 실제 데이터로 확인한다. 이 값이 lookback window를 결정한다.
- [ ] **업스트림 SLA와 내 SLA의 정합을 확인했다.** 소스 완료 시각 + 내 처리 시간 + 재시도 여유 < 내가 약속한 시각이어야 한다. 이 부등식이 성립하지 않는 SLA는 약속이 아니라 희망이다.
- [ ] **타임스탬프 기준을 통일했다.** event time인지 processing time인지, UTC인지 KST인지. 마트 내에서 컬럼마다 기준이 다르면 반드시 사고가 난다. 저장은 UTC, 파티션·표기 기준은 하나로 고정하고 문서화한다. 소스에 서머타임(DST)이 있는 지역 데이터가 섞이면 "하루 23시간·25시간" 문제도 처리 방침을 정한다.
- [ ] **개인정보 컬럼을 소스 단계에서 식별했다.** 마트에 옮긴 뒤 발견하면 이미 늦다. 필요 없는 PII는 아예 가져오지 않는 것이 최선의 통제다.

## 4. 모델링

- [ ] **디멘저널 모델과 OBT(One Big Table) 중 근거를 갖고 선택했다.** 소비자가 BI 도구 중심이고 차원 재사용이 많으면 fct/dim 분리, 소비자가 소수이고 쿼리 단순성이 우선이면 OBT. 유행이 아니라 소비 패턴으로 결정한다.
- [ ] **키 전략을 정했다.** 자연키를 그대로 쓸지, 서로게이트 키(복합키의 해시 등)를 만들지. 서로게이트를 쓴다면 생성 규칙(해시 함수, 컬럼 순서, NULL 처리)을 코드 한 곳에 함수/매크로로 고정한다 — 마트마다 손으로 CONCAT하면 언젠가 어긋난다.
- [ ] **변경되는 차원 속성의 처리 방식을 정했다.** 현재값만 필요하면 SCD Type 1, 이력이 필요하면 Type 2 또는 일별 스냅샷. "나중에 이력이 필요해질 것 같은데"라면 스냅샷이 가장 싸고 단순한 보험이다.
- [ ] **금액·수치 타입을 강제했다.** 금액은 부동소수점(FLOAT)이 아니라 NUMERIC/DECIMAL. 합산이 커질수록 부동소수점 오차는 "왜 1원이 안 맞죠" 문의로 돌아온다. 단위도 통일한다 — 원 단위인지 천원 단위인지, 환율은 어느 시점 스냅샷인지.
- [ ] **NULL 규약을 정했다.** 차원 조인 실패는 NULL로 둘지 `unknown` 기본값으로 채울지, 금액 없음은 NULL인지 0인지. "값이 없음"과 "0"은 다른 사실이다. 팀 규약이 없으면 마트마다 달라지고 소비자가 혼란에 빠진다.
- [ ] **네이밍 컨벤션을 따랐다.** 접두사(`fct_`/`dim_`/`mart_`), 시각 컬럼 접미사(`_at`은 timestamp, `_date`는 date), 불리언 접두사(`is_`/`has_`), 집계 접두사(`cnt_`/`amt_`), 금액 컬럼의 통화 명시(`_krw`, `_usd`). 이름만 보고 타입과 의미를 추론할 수 있어야 한다.
- [ ] **코드값의 디코딩 위치를 정했다.** 상태 코드(`01`, `02`)를 그대로 둘지 레이블(`paid`, `cancelled`)로 풀지. 마트는 소비자를 위한 층이므로 원칙적으로 사람이 읽는 값으로 푼다. 코드 그대로 내보내면 디코딩 로직이 모든 소비자 쿼리에 복제된다.
- [ ] **지표 계산은 마트 한 곳에만 존재한다.** BI 도구에서 지표를 재계산하게 두면 정의가 두 벌이 되고 반드시 어긋난다. 마트가 지표의 single source of truth다.

## 5. 적재 전략

- [ ] **full refresh와 incremental 중 근거를 갖고 선택했다.** 판단 기준: 소스 크기와 스캔 비용, 소스에 update/delete가 있는지, 지각 데이터 범위. 작으면(수 GB 수준) 고민하지 말고 full refresh — incremental은 명백히 필요할 때만 도입하는 복잡도다.
- [ ] **증분 방식을 데이터 특성에 맞췄다.** 파티션 단위 INSERT OVERWRITE는 단순하고 멱등하지만 파티션 전체를 다시 쓴다. 키 기반 MERGE는 부분 갱신이 되지만 스캔 비용과 동시성 문제를 안는다. 지각 데이터가 파티션 경계를 넘는 소스에 파티션 overwrite만 쓰면 과거 파티션이 조용히 틀린다.
- [ ] **멱등성을 보장했다.** 같은 날짜 파라미터로 두 번 실행해도 결과가 같아야 한다. append-only 적재는 재실행 시 중복을 만들므로 마트에서 금지. 스케줄러의 중복 트리거·수동 재실행이 겹쳐도 안전한지(동시 실행 방지 또는 동시 실행 무해)까지 확인한다.
- [ ] **백필을 처음부터 설계했다.** 날짜 범위를 파라미터로 받는 구조로 만든다. "일단 매일 배치만 만들고 백필은 나중에"라고 하면, 나중에 백필용 로직을 따로 짜게 되고 두 로직은 반드시 어긋난다. 전체 백필의 예상 소요 시간과 비용도 미리 계산한다 — 하루 5분짜리 배치가 3년 백필에는 며칠씩 걸릴 수 있다.
- [ ] **lookback window를 적용했다.** 3단계에서 측정한 지각 데이터 분포에 따라 최근 N일을 매일 재처리한다. 지각 데이터가 있는데 당일분만 적재하면 과거 숫자가 조용히 틀린 채 남는다.
- [ ] **스키마 진화 정책을 정했다.** 허용: 컬럼 추가(nullable). 금지: 컬럼 삭제·이름 변경·타입 변경·의미 변경(같은 이름으로 다른 뜻). 금지 항목이 필요하면 새 컬럼/새 버전 테이블로 만들고 마이그레이션 공지를 낸다.
- [ ] **파티셔닝·클러스터링을 소비 쿼리 기준으로 설계했다.** 파티션 키는 소비자가 가장 자주 거는 날짜 필터, 클러스터 키는 그다음 빈도의 필터 컬럼. BigQuery라면 `require_partition_filter`로 전체 스캔을 차단하고, 파티션 개수 한도(테이블당 수천 개 수준)에 걸리지 않는 단위(일별 등)를 확인한다.
- [ ] **감사 컬럼을 넣었다.** `_loaded_at`(적재 시각), `_job_id`(적재 실행 ID) 정도는 기본으로 넣는다. "이 행이 언제 어느 실행으로 들어왔나"를 못 따지는 마트는 장애 조사가 고고학이 된다.

## 6. 데이터 품질 테스트

- [ ] **필수 4종 테스트를 걸었다.** ① 유니크 키 ② 핵심 컬럼 not null ③ 신선도(최신 파티션 존재 여부) ④ row count 급변 감지(전일 대비 임계치, 요일 주기성 감안). 이 4개가 없으면 배포하지 않는다.
- [ ] **소스 대비 reconciliation을 허용 오차와 함께 걸었다.** 마트의 합계(금액, 건수)와 소스의 합계를 비교하는 테스트. 변환 로직 버그는 개별 행 테스트가 아니라 집계 비교에서 잡힌다. 허용 오차(예: ±0.1%)를 명시하지 않으면 정상 범위의 흔들림이 알람 피로를 만들고, 알람 피로는 진짜 알람을 죽인다.
- [ ] **테스트 severity를 구분했다.** 차단(error)과 경고(warn)를 나눈다. 유니크 키 위반은 차단, 참조 무결성 1건 미만 위반은 경고 — 전부 차단으로 걸면 사소한 문제로 SLA를 놓치고, 전부 경고로 걸면 아무도 안 본다.
- [ ] **테스트 실패 시 정책을 정했다.** 경영 지표 마트는 적재 차단이 기본 — 틀린 숫자가 나가는 것보다 늦는 것이 낫다. 차단 시 이전 파티션이 유지되는지(부분 적재로 어중간한 상태가 되지 않는지)도 확인한다.
- [ ] **변환 로직 자체를 단위 테스트했다.** 데이터 테스트(결과 검증)와 별개로, 복잡한 비즈니스 로직(환불 귀속, 환율 적용)은 고정 입력 → 기대 출력의 단위 테스트를 만든다. 데이터 테스트는 "오늘 데이터가 이상한지"를, 단위 테스트는 "로직이 맞는지"를 검증한다 — 서로 대체재가 아니다.
- [ ] **CI 게이트를 걸었다.** 머지 전에 최소한 dry-run(컴파일·권한·문법)과 스키마 diff(의도치 않은 컬럼 변경 감지)가 돌게 한다. "프로덕션에서 처음 실행되는 SQL"을 없애는 것이 목적이다.
- [ ] **실패 알림이 소비자에게도 간다.** 엔지니어만 아는 장애는 소비자 입장에서 "말 없이 틀린 데이터"다. 지연·실패 시 소비자 채널에 자동 공지한다.

## 7. 성능·비용

- [ ] **소비 쿼리의 스캔량을 실측했다.** 1단계에서 받은 최종 쿼리를 마트 위에서 실행해 스캔 바이트를 확인한다. 대시보드가 하루 수백 번 리프레시된다면 이 값이 곧 월 비용이다. 월 비용 추정치(스캔량 × 리프레시 횟수 × 단가)를 산출해서 공유하면 집계 수준·캐시 논의가 빨라진다.
- [ ] **materialization을 판단했다.** 소비 빈도가 낮고 변환이 가벼우면 뷰로 시작한다. 뷰 → 테이블 전환은 쉽지만 반대는 소비자 마이그레이션이 필요하다. "일단 뷰, 느려지면 테이블"이 안전한 기본값이다. 단, 뷰 위에 뷰를 쌓는 체이닝이 3층을 넘으면 디버깅 불능이 되므로 중간을 테이블로 끊는다.
- [ ] **집계 수준을 비용으로 검증했다.** 상세 그레인 마트 하나로 버틸지, 위에 집계 마트를 한 층 더 둘지는 소비 쿼리의 스캔량 실측으로 결정한다.
- [ ] **BI 도구의 캐시·추출 모드를 확인했다.** 같은 마트라도 BI가 라이브 쿼리인지 추출(extract)/캐시인지에 따라 비용과 신선도가 다르다. 마트 SLA와 BI 캐시 주기가 어긋나면 "마트는 갱신됐는데 대시보드는 옛날 숫자" 문의가 온다.
- [ ] **스토리지 비용을 확인했다.** 오래된 파티션의 만료 정책(expiration), 스냅샷·백업의 보존 기간. 매일 쌓는 스냅샷 테이블은 스토리지가 선형으로 늘어난다는 사실을 만들 때 계산한다.

## 8. 운영·거버넌스

- [ ] **오너를 지정했다.** 사람 이름이 아니라 팀·역할로. 문의·장애·변경 요청이 갈 곳이 명확해야 한다.
- [ ] **컬럼 설명과 지표 정의 링크를 채웠다.** 테이블·컬럼 description은 코드(스키마 정의)에 함께 둬서 배포와 같이 굴러가게 한다. 별도 위키는 반드시 낡는다.
- [ ] **변경 관리 규칙을 소비자와 합의했다.** 하위 호환 변경(컬럼 추가)은 무통보 가능, 파괴적 변경(삭제·의미 변경)은 공지 기간(예: 2주) + 마이그레이션 안내 필수. 이 규칙이 없으면 마트는 "무서워서 못 고치는 테이블"이 된다.
- [ ] **PII 컬럼을 식별하고 정책을 적용했다.** 마스킹, 정책 태그, 접근 권한 분리. 마트는 소스보다 접근 권한이 넓어지는 지점이므로 여기서 개인정보가 새기 쉽다.
- [ ] **스케줄 의존성을 명시했다.** 시각 기반 트리거("새벽 3시면 소스가 있겠지")가 아니라 업스트림 완료 이벤트·센서 기반으로 건다. 시각 기반 의존은 업스트림이 늦는 날 반드시 깨진다.
- [ ] **runbook을 만들었다.** 적재 실패 시 재실행 절차, 백필 절차, 흔한 장애 유형과 대응, 에스컬레이션 채널. 새벽에 알람 받은 사람이 만든 사람이 아니어도 대응할 수 있는 수준이면 된다. 한 페이지면 충분하다.
- [ ] **lineage와 사용량 추적을 연결했다.** 이 마트가 어떤 소스에서 오고 어떤 대시보드·마트가 이걸 쓰는지(lineage), 실제로 누가 얼마나 조회하는지(쿼리 로그). 이 둘이 없으면 변경 영향 분석과 폐기 판단이 둘 다 불가능하다.
- [ ] **폐기 경로를 만들었다.** 소비자 등록, 사용량 모니터링, 일정 기간 미사용 시 deprecation 프로세스(공지 → 이름 변경/접근 차단 → 삭제). 마트는 만들 때보다 없앨 때가 훨씬 어렵다 — 없앨 수 있는 구조는 만들 때만 넣을 수 있다.

## 경험칙 요약

운영하면서 비싸게 배운 것들이다.

1. **그레인 문서가 없는 마트는 반드시 중복 집계 사고를 낸다.** 시점은 만든 직후가 아니라 새 소비자가 조인을 시작하는 몇 달 뒤다.
2. **요구사항은 컬럼 목록이 아니라 최종 쿼리로 받는다.** 컬럼 목록으로 시작한 마트는 배포 직후 "이 기준으로도 잘라볼 수 있나요"가 온다.
3. **소스는 문서가 아니라 프로파일링으로 배운다.** 소스의 진짜 스펙은 문서에 없고 데이터에 있다. 빈 문자열 NULL, 테스트 계정, 음수 금액은 항상 있다.
4. **백필은 기능 추가가 아니라 초기 설계다.** 나중에 붙이면 일배치 로직과 백필 로직 두 벌이 생기고, 두 벌은 어긋난다.
5. **incremental은 필요가 증명될 때 도입한다.** full refresh로 충분한 크기인데 incremental로 만든 마트는 복잡도만 지불한다.
6. **틀린 숫자보다 늦은 숫자가 낫다.** 품질 테스트 실패 시 차단을 기본값으로 두고, 예외를 명시적으로 허용한다.
7. **알람은 적게, 대신 전부 진짜로.** 허용 오차 없는 검증과 severity 구분 없는 테스트는 알람 피로를 만들고, 알람 피로는 진짜 사고를 묻는다.
8. **지표 로직 변경은 수정이 아니라 버전 추가다.** 같은 컬럼의 의미를 조용히 바꾸면 과거 대시보드·리포트 전체가 무통보로 오염된다.
9. **첫 배포 후 2주는 검증 기간이다.** 소스 대비 reconciliation을 병행하고, 소비자에게도 검증 중임을 알린다. 배포일에 끝나는 마트 구축은 없다.
10. **마트 개수 증가는 성과가 아니라 부채다.** 신규 구축 요청의 절반은 기존 마트 컬럼 추가로 해결된다.

## Consequences

- 마트 구축의 품질 하한선이 체크리스트로 고정된다. 만드는 사람이 달라져도 그레인 문서·멱등성·품질 테스트는 보장된다.
- 구축 리드타임은 늘어난다. 특히 1~3단계(요구사항·그레인·소스 검증)는 SQL 작성보다 오래 걸릴 수 있다. 이것은 비용이 아니라 운영 사고의 선불이다.
- 체크리스트는 판단을 대체하지 않는다. 각 항목의 "근거를 갖고 선택했다"는 결국 사람이 채워야 하고, 항목을 기계적으로 채우는 것만으로 좋은 마트가 되지는 않는다. 소규모·저위험 마트에는 항목을 의식적으로 생략할 수 있되, 생략을 기록한다.

## Alternatives Considered

1. **문서 없이 리뷰어의 경험에 의존**: 리뷰어가 바뀌면 기준도 바뀐다. 판단 기준이 유실된다 — 미채택.
2. **모델링 방법론 문서(Kimball 등) 링크로 대체**: 방법론은 그레인·SCD 같은 설계는 다루지만 멱등성·백필·SLA·폐기 같은 운영 관점이 없다. 마트 사고의 다수는 운영에서 난다 — 미채택.
3. **플랫폼으로 강제(CI에서 테스트 없으면 머지 차단 등)**: 방향은 맞고 일부(유니크 키·신선도 테스트·스키마 diff)는 자동화 가능하다. 다만 그레인 정의·지표 합의처럼 자동화 불가능한 항목이 핵심이라 체크리스트가 선행돼야 한다 — 부분 채택(향후 CI 게이트로 승격).
