## Context

GA4 분석 사고의 전형적인 형태는 두 가지다. **콘솔 숫자를 전수 데이터로 믿고 결론을 내는 것**, 그리고 **GA4 숫자와 내부 DB 숫자가 다르다며 어느 한쪽을 "틀렸다"고 단정하는 것**. 둘 다 GA4의 수집·처리·보고 구조를 모르는 데서 나온다.

GA4는 UA(Universal Analytics)와 지표 정의부터 다르고, UI 보고서와 BigQuery export의 숫자가 다르며, 동의 모드·광고 차단기·쿠키 수명 때문에 애초에 전수 수집이 아니다. 이 특성들은 버그가 아니라 스펙이다. 스펙을 모른 채 분석하면 "지난달 대비 유저가 줄었다" 같은 보고가 실제로는 태깅 변경이나 threshold 때문일 수 있다.

이 문서는 GA4로 분석 결론을 내기 전에 통과해야 하는 체크리스트다. 데이터 모델의 상세는 [[06_ga4_data_model]] 참고.

## Decision

GA4 기반 분석 결과를 보고·의사결정에 쓰기 전에 아래 체크리스트를 통과시킨다. 특히 0~1단계(수집 신뢰성·지표 정의)를 건너뛴 분석 공유를 금지한다.

## 0. 수집 신뢰성 확인 (분석의 전제)

분석 이전에 수집이 맞는지부터 본다. 태깅이 틀렸으면 이후 모든 단계가 무의미하다.

- [ ] **분석 대상 이벤트의 태깅을 실물로 검증했다.** DebugView·실시간 보고서에서 해당 이벤트가 의도한 시점에 의도한 파라미터와 함께 발화하는지 직접 확인한다. "예전에 누가 심어놓은 이벤트"를 검증 없이 분석하는 것이 사고의 시작이다.
- [ ] **같은 의미의 이벤트가 두 개 이상 아닌지 확인했다.** `sign_up`과 `signup`과 `join_complete`가 공존하는 property는 흔하다. 어느 것이 현행이고 언제 갈아탔는지 모르면 시계열이 성립하지 않는다.
- [ ] **파라미터 타입의 일관성을 확인했다.** 같은 파라미터가 어떤 날은 string, 어떤 날은 int로 들어오면(BQ에서 `string_value`/`int_value`에 나뉘어 저장) 추출 쿼리가 한쪽을 조용히 놓친다. COALESCE로 양쪽을 다 봐야 하는 파라미터인지 실측한다.
- [ ] **중복 발화를 확인했다.** GTM 트리거 중복, SPA 라우팅에서 page_view 이중 발화, 새로고침 재전송. 특히 전환 이벤트의 중복 발화는 전환율을 직접 부풀린다. 이벤트당 유니크 키(transaction_id 등)가 있다면 중복률을 실측한다.

## 1. 지표 정의 확인

- [ ] **이 지표의 GA4 정의를 말할 수 있다.** GA4의 Users는 기본적으로 Active Users다(UA의 Total Users가 아니다). 세션은 UA와 달리 자정에 끊기지 않고, 세션 중 캠페인 소스가 바뀌어도 새 세션이 생기지 않는다. 정의를 모르는 지표로는 결론을 내지 않는다.
- [ ] **key event(전환)의 카운트 방식을 확인했다.** GA4의 전환 수는 UA 목표와 달리 세션당 1회가 아니라 **발생 건수** 기준이다(설정으로 변경 가능). 같은 "전환수"라는 이름으로 UA 시절 숫자·타 도구 숫자와 비교하면 정의 차이만큼 어긋난다.
- [ ] **전환율의 분모를 합의했다.** 세션 기준인지, 유저 기준인지, 특정 이벤트(노출) 기준인지. "전환율 3%"는 분모 없이는 숫자가 아니다. GA4 UI의 session conversion rate와 user conversion rate는 별개 지표다.
- [ ] **reporting identity 설정을 확인했다.** blended / observed / device-based 중 무엇인지에 따라 유저 수가 달라진다. 설정을 바꾸면 **과거 보고서 숫자도 바뀌므로**, 시계열 비교 중에 설정이 바뀌었는지도 본다.
- [ ] **UA 시절 숫자와 직접 비교하지 않는다.** Users·세션·이탈률·전환수 전부 정의가 다르다. "작년(UA) 대비 세션 증감" 같은 비교는 무의미하며, 하려면 정의 차이를 명시한 각주가 필수다.
- [ ] **UI 기준인지 BigQuery 기준인지 명시했다.** 같은 "일별 Users"도 UI는 HLL 기반 근사치, BigQuery에서 직접 센 값은 exact count다. 두 숫자는 원래 다르다. 보고서에는 어느 쪽 기준인지 한 줄을 반드시 넣는다.

## 2. UI 보고서의 함정 점검 (탐색·표준 보고서를 쓸 때)

- [ ] **샘플링 여부를 확인했다.** 탐색(Explorations) 보고서는 쿼리가 이벤트 한도(표준 property 1,000만)를 초과하면 샘플링된다. 보고서 좌상단의 샘플링 아이콘을 확인하고, 샘플링된 결과로는 작은 차이를 논하지 않는다. 기간을 쪼개서 한도 아래로 내리면 샘플링을 피할 수 있는 경우가 많다.
- [ ] **(data) threshold를 확인했다.** Google signals가 켜진 property는 소수 사용자 행이 숨겨진다. 세그먼트 합계가 전체와 안 맞는 가장 흔한 이유다. 행이 숨겨진 보고서로 "이 세그먼트는 0이다"라고 결론 내면 안 된다.
- [ ] **(other) 행을 확인했다.** 고카디널리티 차원(페이지 경로, 커스텀 파라미터)은 카디널리티 한도를 넘으면 "(other)"로 뭉개진다. (other)가 상위권에 있으면 그 차원 분석은 UI에서 하지 말고 BigQuery로 간다.
- [ ] **세그먼트·비교의 스코프를 확인했다.** 세그먼트에는 유저 스코프("이 행동을 한 적 있는 유저의 모든 세션")와 세션 스코프("이 행동이 일어난 세션만")가 있다. 스코프를 착각하면 같은 조건으로 전혀 다른 모수를 보게 된다.
- [ ] **데이터 완성 시점을 지켰다.** GA4 처리 지연은 24~48시간이고, late hit은 최대 72시간까지 반영된다. 어제·오늘 데이터로 추세 판단하지 않는다. 주간 비교는 완성된 주끼리만 한다.
- [ ] **커스텀 디멘션 등록 여부와 등록일을 확인했다.** 커스텀 파라미터는 커스텀 디멘션으로 등록해야 UI에 보이고, **등록 이전 데이터는 소급되지 않는다.** "이 파라미터 데이터가 6월부터만 있다"는 수집 시작일이 아니라 등록일일 수 있다. BQ에는 등록과 무관하게 수집 시점부터 있으므로 교차 확인이 가능하다.
- [ ] **데이터 보관 기간을 확인했다.** retention 설정(표준 최대 14개월)은 탐색 보고서에만 적용된다. 탐색에서 작년 데이터가 안 나오는 것은 데이터가 없어서가 아니라 보관 설정 때문일 수 있다 — 장기 시계열은 처음부터 BQ로 간다.

## 3. 수집 계층 점검

- [ ] **분석 기간에 수집 변경이 있었는지 확인했다.** 태깅 배포, GTM 컨테이너 버전, 동의 모드 도입, 스트림 추가, 향상된 측정 토글, 커스텀 디멘션 등록. 시계열의 계단은 사용자 행동 변화보다 수집 변경인 경우가 더 많다. 수집 변경 로그(annotation)가 없는 property는 시계열 분석의 전제가 없는 것이다 — GTM 컨테이너 버전 이력이 최후의 감사 로그다.
- [ ] **웹·앱 스트림의 시맨틱 차이를 인지했다.** 같은 property의 웹(gtag/GTM)과 앱(Firebase SDK)은 자동 수집 이벤트 목록도, 세션 처리도, 도착 지연 분포도 다르다(앱은 오프라인 큐 배치 업로드). 스트림 합산 지표는 이 차이를 안고 있는 숫자다. 스트림별로 나눠서 먼저 보는 것이 기본이다.
- [ ] **봇·내부 트래픽 필터를 확인했다.** 내부 IP 필터(`traffic_type` 파라미터 + 데이터 필터), 개발·스테이징 환경의 이벤트가 프로덕션 property로 섞여 들어오는지. 릴리스 직후 트래픽 급증의 단골 원인이다. **데이터 필터는 적용 시점 이후만 유효하고 소급 제거되지 않는다** — 필터 적용일 전후의 시계열은 연속이 아니다.
- [ ] **cross-domain·referral 설정을 확인했다.** 자사 도메인 간 이동에 cross-domain 측정이 안 걸려 있으면 세션이 끊기고 self-referral이 생긴다. 결제 PG·본인인증 도메인이 unwanted referral 목록에 없으면 구매 직전 유입 채널이 전부 PG사로 잡힌다 — 전환 채널 분석을 하기 전에 반드시 확인한다.
- [ ] **GA4는 전수가 아님을 전제했다.** 광고 차단기, ITP의 쿠키 수명 제한, 동의 거부(Consent Mode)로 일부 트래픽은 애초에 수집되지 않는다. 동의 모드에서 behavioral modeling이 켜져 있으면 UI 숫자에는 모델링(추정) 데이터가 섞이고, 이 모델링 데이터는 BigQuery export에는 없다 — UI·BQ 불일치의 또 다른 원인이다. 동의 거부율이 높은 시장(EU) 데이터일수록 이 간극이 크다.

## 4. BigQuery export로 분석할 때

- [ ] **일별 테이블(events_YYYYMMDD)의 확정 시점을 기다렸다.** intraday 테이블은 잠정치다. 일별 테이블도 late hit 반영으로 최대 72시간까지 갱신될 수 있다. 파이프라인이 GA4 export를 소스로 쓰면 lookback 재처리를 설계에 넣는다.
- [ ] **`_TABLE_SUFFIX`로 기간을 잘랐다.** GA4 export는 파티션 테이블이 아니라 날짜 샤딩 테이블이다. `events_*`에 SUFFIX 필터 없이 쿼리하면 전체 기간 풀스캔 — GA4 분석 비용 사고의 1순위 원인이다.
- [ ] **세션을 올바르게 재구성했다.** `ga_session_id`는 세션 시작 시각의 epoch 초라서 단독으로 유니크하지 않다 — 반드시 `user_pseudo_id`와 조합한다. 세션 수를 셀 때 이걸 틀리면 조용히 부풀거나 준다.
- [ ] **타임존을 통일했다.** `event_timestamp`는 UTC 마이크로초, `event_date`는 property 타임존 기준 — **같은 행 안에서 기준이 다르다.** UI와 맞추려면 `event_date`(또는 KST 변환)로, UTC 기준이 필요하면 timestamp로. 어느 쪽인지 쿼리에 명시한다.
- [ ] **트래픽 소스 필드를 올바르게 골랐다.** `traffic_source.*`는 유저의 **최초 획득** 소스다(세션 소스가 아니다). 세션 단위 채널 분석은 `session_traffic_source_last_click.*`, 이벤트 수집 시점 원본은 `collected_traffic_source.*`. 이 셋을 혼동하면 채널 숫자가 그럴듯하게 틀린다.
- [ ] **user_pseudo_id의 한계를 전제했다.** 브라우저·디바이스 단위 식별자이고 쿠키 삭제·ITP로 리셋된다. "유저 수"는 실제 사람 수보다 부풀려진 값이다. 동의 거부 트래픽은 `user_pseudo_id`가 NULL로 들어올 수 있으므로 COUNT(DISTINCT) 시 NULL 처리와 제외 건수를 명시한다. 사람 단위 분석이 필요하면 User-ID 수집이 선행돼야 한다.
- [ ] **중복 이벤트를 확인했다.** 클라이언트 재전송으로 완전 동일 이벤트가 중복 유입될 수 있다. 전환·매출 집계 전에 (user_pseudo_id, event_timestamp, event_name, event_bundle_sequence_id) 수준의 중복률을 실측하고, transaction_id가 있는 이벤트는 그것으로 dedup한다.
- [ ] **원본 직접 쿼리 대신 평탄화 층을 썼다.** 자주 쓰는 파라미터를 컬럼으로 풀어둔 staging 테이블(dbt 등)을 한 층 두고 그 위에서 분석한다. 분석마다 UNNEST를 손으로 쓰는 조직은 쿼리 비용과 추출 실수(타입 분기 누락)를 같이 지불한다.
- [ ] **export 한도를 확인했다.** 무료 property의 daily export는 일 100만 이벤트 한도가 있다. 한도 초과로 조용히 잘린 날이 있는지 일별 이벤트 수 추이로 감시한다.

## 5. 어트리뷰션·채널 분석

- [ ] **리포트의 어트리뷰션 스코프·모델을 확인했다.** GA4에는 event-scoped와 session-scoped 트래픽 소스가 공존하고, 리포트마다 쓰는 스코프가 다르다. 전환 어트리뷰션 기본값은 데이터 기반(DDA)이라 last click 기준 채널 숫자와 다르고, 어트리뷰션 lookback window 설정(전환 기준 30/90일 등)에 따라서도 달라진다. 어느 스코프·모델·윈도우의 숫자인지 모르면 채널 비교를 하지 않는다.
- [ ] **채널 그룹은 UI 파생 차원임을 전제했다.** Organic Search, Paid Social 같은 채널 그룹은 source/medium 위에 Google의 분류 규칙을 얹은 것이고 BQ에는 없다. BQ에서 채널 분석을 하려면 분류 규칙을 직접 구현해야 하며, UI와 완전히 같아지지는 않는다.
- [ ] **UTM 규약이 있는지 확인했다.** utm_source/medium의 대소문자·표기 규약이 없으면 같은 채널이 여러 행으로 흩어진다. 채널 분석은 태깅 규약이 선행 조건이다. 단축 URL·앱 딥링크·QR 경유 시 파라미터가 살아남는지도 실측한다.
- [ ] **Google Ads 캠페인 분석은 조인이 필요함을 전제했다.** 자동 태깅(gclid)은 UI에서는 캠페인으로 풀리지만 BQ에는 gclid 문자열만 온다. BQ에서 캠페인 단위 성과를 보려면 Google Ads 데이터와 별도 조인해야 한다.
- [ ] **direct와 (not set)의 증가를 채널 성과로 읽지 않았다.** 리다이렉트 체인의 파라미터 유실, 앱 전환, self-referral 설정 누락, 동의 거부 등 수집 문제의 지표인 경우가 많다. direct 비율의 추이는 채널 지표가 아니라 **수집 건강도 지표**로 본다.

## 6. 결론을 내기 전에

- [ ] **세그먼트의 표본 크기를 확인했다.** 유저 50명짜리 세그먼트의 전환율 2배 차이는 인사이트가 아니라 노이즈일 확률이 높다. 비율 지표는 분자·분모의 절대 수를 함께 표기한다.
- [ ] **외생 변수를 대조했다.** 프로모션, 앱 릴리스, 마케팅 집행, 계절성, 공휴일. 지표 변동을 제품 요인으로 결론 내기 전에 같은 기간의 이벤트 캘린더를 옆에 놓고 본다.
- [ ] **차이가 결론을 바꿀 크기인지 판단했다.** GA4 숫자에는 구조적 오차(수집 누락, 근사, 모델링)가 항상 깔려 있다. ±수 퍼센트 수준의 차이로 의사결정을 뒤집지 않는다.
- [ ] **내부 DB와의 차이는 "차이율의 안정성"으로 본다.** 결제 DB 매출과 GA4 매출은 원래 안 맞는다(수집 누락, 환불 시점, 중복 전송). 절대값 일치를 목표로 하지 말고 차이율을 추적해서, 차이율이 갑자기 움직일 때만 조사한다.
- [ ] **탐색 중 발견한 패턴은 결론이 아니라 가설로 보고했다.** 데이터를 뒤지다 찾은 흥미로운 패턴은 다중 비교의 산물일 수 있다. "발견했다"가 아니라 "가설이 생겼다, 다음 기간/실험으로 검증하겠다"로 쓴다.
- [ ] **재현 쿼리를 남겼다.** UI 스크린샷 기반 보고는 재현이 안 된다. BigQuery 분석이면 쿼리를, UI 분석이면 보고서 설정(기간·세그먼트·필터·어트리뷰션 설정)을 결과와 함께 남긴다.

## 경험칙 요약

1. **GA4는 회계 장부가 아니라 행동 추세계다.** 방향·추세·비율 분석에 쓰고, 정산·매출 확정 같은 전수 정확성이 필요한 곳에는 처음부터 내부 DB를 쓴다.
2. **태깅 검증 없는 분석은 정교한 쓰레기다.** 분석 역량보다 수집 신뢰성이 먼저다. DebugView 10분이 잘못된 결론 한 달을 막는다.
3. **UI와 BigQuery 숫자가 다른 것은 버그가 아니라 스펙이다.** 근사(HLL), threshold, 모델링 데이터, 타임존, identity 통합 — 불일치의 원인은 항상 이 다섯 중에 있다.
4. **시계열의 계단은 사람보다 태깅이 만든다.** 급증·급감을 보면 사용자 행동 가설보다 수집 변경 이력을 먼저 뒤진다. 수집 변경 로그가 없으면 지금부터라도 남긴다.
5. **커스텀 디멘션 등록은 소급되지 않는다.** 파라미터를 심는 날 등록까지 같이 한다. "나중에 등록"의 비용은 그 사이 기간의 데이터 전체다.
6. **두 시스템의 숫자는 맞추는 게 아니라 차이율을 감시하는 것이다.** 절대값 일치에 쓰는 시간은 대부분 낭비고, 차이율의 급변만이 신호다.
7. **세션 조인 키는 (user_pseudo_id, ga_session_id)다.** ga_session_id 단독 조인은 GA4 BigQuery 분석의 가장 흔한 실수다.
8. **`traffic_source`는 세션 소스가 아니다.** 채널 분석 오류의 절반은 user-scoped 최초 획득 소스를 세션 소스로 착각한 데서 나온다.
9. **direct의 증가는 성과가 아니라 증상이다.** 수집 설정(cross-domain, referral 제외, UTM 유실)부터 의심한다.
10. **분석 요청을 받으면 숫자보다 정의를 먼저 합의한다.** "전환율"이 세션 기준인지 유저 기준인지, 분모가 뭔지 합의 없이 뽑은 숫자는 반드시 재작업이 된다.
11. **작은 세그먼트의 극적인 차이는 대부분 표본 노이즈다.** 비율에는 항상 절대 수를 붙인다.

## Consequences

- 분석 착수 전 확인 비용이 늘어난다(태깅 검증, 수집 변경 이력, 지표 정의, 데이터 완성 시점). 대신 "숫자가 왜 다르죠"로 시작하는 재작업과 신뢰 손실을 구조적으로 줄인다.
- GA4 UI 숫자의 한계를 명시하는 보고는 "그래서 이 숫자를 믿어도 되나"라는 질문을 유발할 수 있다. 이는 회피할 문제가 아니라 GA4를 쓰는 조직이 한 번은 합의해야 하는 문제다.
- 0단계(수집 신뢰성)는 분석가 혼자 통과시킬 수 없는 항목이 있다 — 태깅 QA·수집 변경 로그는 개발·마케팅과의 공동 규율이다. 이 체크리스트는 그 합의를 요구하는 문서이기도 하다.
- GA4 스펙(한도, threshold, export 정책)은 Google이 수시로 바꾼다. 이 문서의 구체 수치는 `valid_from` 기준이며, 주기적 리뷰가 필요하다.

## Alternatives Considered

1. **UI 보고서만으로 분석**: 빠르고 접근성이 좋지만 샘플링·threshold·(other)·카디널리티 한계를 통제할 수 없다. 탐색적 확인에는 쓰되 의사결정용 숫자는 BigQuery 기준으로 — 부분 채택.
2. **자체 이벤트 트래킹으로 전면 대체**: 수집을 완전히 통제할 수 있지만 SDK·수집 서버·파이프라인 구축 비용이 크고, 광고 플랫폼 연동(전환 피드백)은 결국 GA4/gtag가 필요하다. 규모가 커지면 병행이 현실적 — 미채택(현 단계).
3. **서드파티 분석 도구(Amplitude, Mixpanel 등)로 대체**: 이벤트 분석 UX는 우수하나 비용이 들고, GA4의 함정(식별자, 동의, 수집 누락)의 상당수는 클라이언트 수집 기반 도구 공통의 문제라 도구 교체로 사라지지 않는다 — 미채택.
